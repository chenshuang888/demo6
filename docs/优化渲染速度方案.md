 新链路（你的思路）

  T+0ms      用户点 calc → 不切屏，menu 仍在显示
             page_dynamic_app_prepare("calc") 被调
             ├─ 创建 off-screen screen（lv_obj_create(NULL)）
             ├─ 创建 top_bar / list_root（off-screen 上）
             ├─ dynamic_app_ui_set_root(off-screen list_root)
             └─ dynamic_app_start("calc")            （脚本异步开跑）

  T+5ms      脚本 eval calc.js
  T+50ms     脚本 rerender → 60+ 条 cmd 入 queue
  T+60-90ms  UI Task drain → 在 off-screen 上构建 LVGL 对象树
             ⬆ 期间 LVGL 还在持续 render menu 旧屏，用户毫无感觉
  T+90ms     脚本调 sys.ui.attachRootListener("appRoot")
             → UI Task 处理这条 cmd → 触发 on_ready
             → page_router_switch(PAGE_DYNAMIC_APP)
                ├─ destroy menu (lv_obj_del 旧 screen)
                └─ create 命中已 prepared 的 screen，直接 lv_scr_load
  T+91ms     用户屏幕：menu 一帧后切到 **完整的** calc，零中间态

  关键设计点（确认下）

  1. "准备完成"的信号用 attachRootListener：所有 4 个 app 末尾都恰好调它，是天然的 build 完成标记。不用加 native 也不用改 JS。
  2. 超时兜底 800ms：万一脚本崩或漏调 attachRootListener，到时间强切，避免卡死在 menu。
  3. menu 在 prepare 期间冻结输入：避免用户疯点导致重复 prepare。简单做法：prepare 启动后 set 一个 busy flag，菜单 click 直接
  return。
  4. 改动只在 page_dynamic_app + ui drain + page_menu 三处，page_router 完全不动。其他原生页保持原行为（同步切换，本来就快）。
  5. 不改 JS —— 4 个 app 一行不动。
