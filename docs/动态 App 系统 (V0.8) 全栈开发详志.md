🛠️ 动态 App 系统 (V0.8) 全栈开发详志日期： 2026-04-27版本状态： V0.8.5 "真·动态化" 达成一、 存储层：从只读到“LittleFS”持久化为了摆脱“更新 App 必须烧录固件”的窘境，我们将存储架构进行了彻底重构：分区表变更：将 storage 分区 subtype 从 spiffs 改为 0x83 (LittleFS)。LittleFS 具备掉电保护，在无线传输中途断电时能有效防止文件系统损坏。双源 Registry 检索：内嵌优先：系统启动时先检索固件自带的 7 个核心 App（如闹钟、计算器），确保基本功能不受文件系统状态影响。FS 增量：若内嵌未命中，则自动在 /littlefs/apps/ 目录下搜索对应的 .js 文件。原子化写入方案：采用 .tmp 文件缓存 + rename 重命名机制。只有当整个文件通过 CRC32 校验后，才会覆盖旧文件，确保更新过程的“全有或全无”。二、 传输层：解决“50ms vs 500ms”的异步之战这是本次开发最硬核的逻辑点：如何让 BLE 回调不被耗时的 Flash 写入卡死。1. 性能标准分析操作类型典型延迟线程容忍度决策NVS 写入~20msBLE 回调 (50ms 红线)勉强直接调LittleFS 写入200-500msBLE 回调 (50ms 红线)必须异步化2. 异步 Manager 架构BLE Service (a3a40001)：负责 GATT 交互。其 access_cb 只负责将数据包 push 进队列（耗时 < 1ms），然后立即返回，绝不阻塞 NimBLE 主任务。Upload Manager：拥有专属的后台任务 (Consumer Task)，以 Pri=2、4KB Stack 独立运行。它负责从队列中取出分片，缓慢地进行 fwrite 操作。状态机协议：通过 START（声明长度与 CRC）、CHUNK（分片传输）、END（触发提交）三段式协议，实现流式上传。三、 JS 生态：标准库 prelude.js 的诞生为了精简 App 代码并修复黑屏 Bug，我们提取了 JS 标准库：架构精简：从 7 个 App 中抽离出 348 行通用的 VDOM 框架与 BLE 通信逻辑，将其封装为固件内嵌的 prelude.js。代码瘦身：业务脚本行数从 3431 行精简至 1940 行，净减少 1491 行 冗余代码。关键 Bug 修复：Catch 变量冲突：修复了精简版 QuickJS 引擎将 catch(e) 变量提升导致的 SyntaxError（该 Bug 曾导致 Weather/Music App 无法启动）。事件分发失效：在 prelude.js 中自动完成 setDispatcher 注册，解决了精简 App 无法响应按钮点击的问题。四、 PC 端工具链：CustomTkinter 管理器配套开发的 GUI 工具彻底改善了开发体验：双线程桥接：基于 AsyncioBridge 架构，让 Bleak 的异步 BLE 操作与 Tkinter 的阻塞式 UI 循环和谐共存。确认机制：每发送一帧数据都会等待固件回传 Status Notify，通过序列号 (seq) 匹配实现精准的流控与重传。管理功能：支持 App 列表刷新、远程删除以及实时日志监控。当前工程统计：新增 Service: dynapp_upload_service (a3a40001)。核心 Store: dynapp_script_store.c（负责 FS 与 Flash 交互）。开发效率: 从“修改->编译->烧录(3min)”缩短为“保存->GUI上传(1s)->运行”。
