● 勘察完了。发现一个关键事实：如果做成独立 component，会撞循环依赖。

  依赖关系

  notify_manager.c     → persist.h            (manager 要 NVS)
  *_service.c          → *_manager.h          (service 要 push)

  - manager 想独立 → 需要 persist，但 persist 在 services → manager REQUIRES services
  - services/*_service.c 又得 push 到 manager → services REQUIRES manager
  - → 循环

  两个可行方案

  方案 A（推荐，最轻）：manager/ 作为 services 的子目录，不单独拆 component

  services/
  ├── *_service.c/h       (5 对，留)
  ├── ble_conn.c/h        (留)
  ├── persist.c/h         (留)
  ├── settings_store.c/h  (留)
  ├── manager/            ← 新建子目录
  │   └── *_manager.c/h   (5 对挪进来)
  └── CMakeLists.txt      (SRCS 路径改 "manager/xxx.c" + INCLUDE_DIRS 加 "manager")

  - 只有文件物理位置变，源码一行不改（所有 #include "xxx_manager.h" 继续生效）
  - 工作量：mv 10 个文件 + 改 1 个 CMakeLists.txt
  - 风险：几乎为零

  方案 B（彻底，代价大 10x）：manager/ 做成平级顶层 component

  需要同步做：(1) 把 persist + settings_store 拆成新的 storage component（否则循环依赖解不开）；(2) 改顶层 CMakeLists.txt 的
  EXTRA_COMPONENT_DIRS；(3) 3 个 component 的 REQUIRES 关系。

  但用户明确说过 "settings_store、persist 先留着"——跟方案 B 的前置条件冲突。

  我的建议

  services/
  ├── (8 个 .c/.h)
  └── manager/  (10 个 .c/.h)

  具体步骤

  1. mkdir services/manager
  2. mv services/{time,weather,notify,media,system}_manager.{c,h} → services/manager/

● 核心区别是有没有内部状态需要初始化。

  对比三个 storage 的"内部状态"

  ┌───────────────────┬────────────────────────────────┬──────────────────────────────────┬────────────────────────────────┐
  │      Storage      │        内部 static 变量        │               作用               │         所以需要 init          │
  ├───────────────────┼────────────────────────────────┼──────────────────────────────────┼────────────────────────────────┤
  │ backlight_storage │ s_backlight（内存缓存）        │ get() 从缓存返回，避免每次读 NVS │ ✅ init 时把 NVS 值加载进缓存  │
  ├───────────────────┼────────────────────────────────┼──────────────────────────────────┼────────────────────────────────┤
  │ time_storage      │ s_last_save_us（上次保存时刻） │ tick_save 用它做 5 分钟防抖      │ ✅ init 时记录启动时刻作为基准 │
  ├───────────────────┼────────────────────────────────┼──────────────────────────────────┼────────────────────────────────┤
  │ notify_storage    │ 无任何 static 变量             │ —                                │ ❌ 无状态可初始化              │
  └───────────────────┴────────────────────────────────┴──────────────────────────────────┴────────────────────────────────┘

  看 notify_storage.c 你会发现只有两个函数体，每次调用都是直接转发给 persist，中间不缓存、不防抖、不记状态——属于"无记忆的透传层"。

  为什么三个的设计不对称？

  因为各业务的缓存/防抖策略放在哪一层不同：

  ┌──────┬────────────────────────────────────┬──────────────────────────────────────────────┐
  │ 业务 │            内存缓存在哪            │                 防抖决策在哪                 │
  ├──────┼────────────────────────────────────┼──────────────────────────────────────────────┤
  │ 背光 │ backlight_storage.c 的 s_backlight │ set() 立即写（背光不频繁）                   │
  ├──────┼────────────────────────────────────┼──────────────────────────────────────────────┤
  │ 时间 │ 无需缓存（只启动时读 1 次）        │ time_storage.c 里 5 分钟间隔                 │
  ├──────┼────────────────────────────────────┼──────────────────────────────────────────────┤
  │ 通知 │ notify_manager.c 的 s_ring[10]     │ notify_manager.c 的 tick_flush dirty+2s 防抖 │
  └──────┴────────────────────────────────────┴──────────────────────────────────────────────┘

  注意最后一行：通知的"环形缓冲 + 防抖"逻辑留在业务层 notify_manager 了——因为环形缓冲涉及 head 指针、版本号、count
  边界这些业务语义，下沉到存储层不合适。

  所以 notify_storage 是故意设计成无状态的薄 wrapper——它就干一件事：隐藏 NS_NOTIFY / KEY_NOTIFY_RING 这两个常量，让上层调
  notify_storage_load/save 而不是直接调 persist_get/set_blob。

  换句话说

  - 背光/时间的"缓存 + 写策略"下沉到了 storage（数据简单，策略通用）
  - 通知的"缓存 + 写策略"留在了 manager（数据复杂，策略有业务语义）

  这种分工是合理的，不是 bug。如果硬给 notify_storage 塞个 init，它没事可做——prepload blob 会导致 1360 字节双重缓存（storage 一份 +
  manager 一份），反而浪费 PSRAM。

  判断依据：一个模块需不需要 init，看它有没有启动时必须确定的内部状态。没有就不需要，强行加 init 是过度设计。
