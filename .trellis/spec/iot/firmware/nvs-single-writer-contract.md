## 适用范围

---

## 一句话原则

**运行期 NVS 读写只能发生在单一“写入上下文”（单写者任务）中；其它线程只能投递请求，不能直接碰 NVS。**

---

## 语义约束（不变量）

### 不变量 1：NVS 写入集中在固定线程上下文

对话中的稳定落地形态是：

- 启动期：`app_main`（初始化与加载）
- 运行期：`ui_task`（周期逻辑 + LVGL 事件回调）

### 不变量 2：通信线程不直接访问 NVS

典型规则：

- BLE host 线程 **绝不直接碰 NVS**
- 类似 `notify_manager_push` 只做 `xQueueSend`（或 `xTaskNotify`），把数据交给单写者任务

### 不变量 3：运行期写入串行化

对话中给出的串行保证方式是：

`ui_task` 每轮依次执行（示例）：

- `process_pending`
- `tick_flush`
- `tick_save_time`
- `lvgl_port_task_handler`

因此 `persist_set_blob` / `persist_set_i64` 不会并发；同任务内的事件回调天然互斥。

### 不变量 4：启动期与运行期不重叠

`persist_init/*_init/load_*` 在创建 `ui_task` 之前完成，避免竞争窗口。

## 实现建议（保持语义不变，按需要演进）

### 默认实现：UI 任务既渲染又落盘（负载低时够用）

优点：简单、语义清晰、不会过度设计。

### 何时需要拆 `persist_task`（仍然单写者）

对话中给出的触发信号：

1. **可见卡顿**：某些落盘时刻动画掉帧
2. **日志信号**：落盘相关警告/阻塞迹象增多
3. **落盘数据变大**：blob 规模增大（例如 >4KB）或落盘频率明显提高

拆分原则：

- 把 `tick_flush` / `tick_save_time` 挪到低优先级 `persist_task`
- 触发方式用 `xTaskNotify` / 信号量 / 队列传“请落盘”信号
- **消费者仍然只有这一个任务**，single-writer 语义不破坏

---

## 验收标准

- 任意通信线程（BLE host 等）代码路径中，不出现对 `persist_*` / NVS API 的直接调用
- 运行期所有 NVS 写入都有唯一入口（单写者任务）
- 压力下不会出现“偶发丢数据/脏数据/写入冲突”类问题（至少能通过日志定位到唯一写入通道）
