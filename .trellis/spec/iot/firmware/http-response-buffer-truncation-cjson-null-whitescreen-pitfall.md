# Pitfall：HTTP 响应缓冲区太小 → JSON 被截断 → `cJSON_Parse` 返回 NULL → 白屏/崩溃

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo3.md`）里对 `app_novel` 的排查：真实响应 2520B > 设备缓冲 2048B，末尾 472B 被丢弃，导致 JSON 不完整，最终解析失败。

## 症状

- ESP32 端发起 HTTP 请求后：
  - UI 白屏/不刷新
  - 日志显示 `cJSON_Parse` 返回 NULL（或后续字段读取崩溃）
- 现象可能“看起来随机”：
  - 小响应正常
  - 响应稍大就必现

---

## 根因（典型实现的隐性假设）

很多实现会在 HTTP 回调里做“尽量往固定 buf 里塞”：

- `copy = min(len(chunk), avail)`
- 多出来的部分直接丢弃

一旦响应总长度 > buf_max：

- 后半段 JSON 丢失
- 解析必失败（`cJSON_Parse` 需要完整 JSON）

---

## 修复策略（按推荐顺序）

### 方案 A：把 payload 设计得“天然小”（优先）

从系统设计上避免大 JSON：

- 服务器端只返回设备真正需要的字段
- 限制列表条数/每页大小
- 字符串字段限制长度（并在服务端做截断/省略）

优点：从根上降低内存和延迟压力。

### 方案 B：设备端按 `Content-Length` 动态分配（更通用）

思路：

- 请求前/收到 header 后读取 `Content-Length`
- 若长度超过上限：
  - 直接报错并提示（不要继续解析）
- 若可接受：
  - 按长度在 PSRAM 分配缓冲
  - 完整读取后再 `cJSON_Parse`

### 方案 C：临时加大固定缓冲（只作为止血）

对话里出现过的止血修复是：

- `2048 → 4096` 后测试全绿

风险：

- 你只是在推迟崩溃阈值
- 下一次服务端字段稍增，又会复现

---

## 预防：把“溢出”变成测试用例

对话里的有效做法：

- 在主机侧获取真实响应字节数（UTF-8 bytes）
- 对比设备侧 `BUF_SIZE`
- 直接 fail（比“烧录后白屏”省时间）

---

## 验收标准

- 服务端返回更大数据时：
  - 设备端能明确报错并回退（而不是白屏）
- 设备端不会因为响应超过固定 buf 就“悄悄丢尾巴”
- `cJSON_Parse` 失败时有兜底 UI（提示/重试/返回上一页）
