# Checklist：FreeRTOS 任务架构“味道”快速自检（优先级 / 栈 / 同步 / 分层）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32.md`）中的“ESP32 固件架构深度分析报告”里对典型问题的归纳（任务过多、优先级同质化、共享资源无保护、跨层调用等）。

---

## 目标

用一张清单快速发现会在中后期爆炸的固件结构问题，避免：

- “能跑，但偶发卡顿/掉线/看门狗重启”
- “加一个功能就要改 5 个 task”
- “排查一次问题必须全量烧录+长时间复现”

---

## 优先级（Priority）味道

- [ ] 所有任务优先级都一样（例如都为 5），且没有明确的“关键链路优先级更高”的理由
- [ ] 网络接收/解码/显示/控制输入都挤在同一优先级，导致关键任务无法及时响应
- [ ] 没有把“低频后台任务”降级（例如周期上报/扫描/日志整理）

建议动作：

- 先画出关键链路（例如：网络帧接收→解析→渲染 / 输入→回传），只让关键链路拥有更高优先级
- 其他任务默认低优先级 + 事件驱动（而不是硬循环）

---

## 栈（Stack）味道

- [ ] 栈大小按“拍脑袋”分配：简单任务也给 2KB/4KB，大任务给到 16KB 仍不确认是否必要
- [ ] 存在少量任务异常巨大（例如扫描/解析任务 16KB），但没有拆分/复用缓冲策略
- [ ] 没有任何“栈水位”或 OOM/栈溢出定位手段

建议动作：

- 先用最小闭环跑起来后，按实际峰值调栈（而不是先把所有栈拉满）
- 把大缓冲从栈迁移到静态/堆，并明确所有者（单写者优先）

---

## 同步与通信（Sync/IPC）味道

- [ ] 只有 mutex（例如只保护发送），没有 queue/event group/通知，导致共享状态靠“全局变量 + 轮询”
- [ ] 多个任务读写同一连接/同一状态机，但没有明确的所有者（owner）
- [ ] 断线重连逻辑分散在多个任务里（谁都能重连，最终谁都不可靠）

建议动作：

- 给每条“有状态资源”指定 owner task（socket、Wi‑Fi 状态、播放器状态、UI 状态）
- 其他任务一律通过 queue/notify 发请求，禁止直接改状态机

---

## 分层（Layering）味道

- [ ] App/task 代码直接调用 BSP/HAL（跨过 Core/Service 层）
- [ ] Core 层声明了接口但未实现，真正实现散落在 App/task（导致后续无法复用/测试）
- [ ] GPIO/外设操作散落在网络任务里（例如 WebSocket/HTTP task 里直接点灯）

建议动作：

- 把硬件操作收敛到驱动/服务模块，task 只做调度与状态机推进
- Core 层必须“可复用”：接口 + 最小实现 + 可替换（mock）边界

---

## 验收标准（自检通过的最低要求）

- 能画出关键链路并解释“为什么这些任务优先级更高”
- 每个有状态资源都有唯一 owner，其他模块只能通过消息请求
- 栈大小有依据（至少能说明：为什么需要/不需要 16KB）
