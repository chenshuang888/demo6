## 一句话结论

当用户输入很快时，请求会乱序返回：**用 `seq`（请求序列号）保证“只接收最新响应”**，其余全部丢弃。

---

## 典型问题（为什么必须做）

用户输入：`n` → `ni` → `nih`，你可能发出 3 个候选请求。

网络/主机处理延迟会导致：

- `nih` 的响应先到（正确）
- `n` 的响应最后到（过期）

如果不处理，候选栏会“倒退”回旧结果，用户会误选，体验直接崩。

---

## 推荐实现（对话里的最小有效方案）

### 1) 请求结构体带 `seq`

对话里把 `/pinyin` 和 `/suggest` 统一成一类“候选请求”，用同一个 task 去处理，只是 path/key/value 不同：

```c
typedef struct {
    char path[16];    // "/pinyin" 或 "/suggest"
    char key[16];     // "pinyin" 或 "context"
    char *value;      // 拼音字符串 或 输入框完整内容
    uint32_t seq;     // 请求序列号（关键）
} cand_req_t;
```

### 2) 全局保存“最新请求序列号”

- `s_req_seq`：每次发请求都自增，并写成“当前最新”
- 发请求时把 `seq` 拷贝进 `cand_req_t`

### 3) 解析响应时丢弃过期结果（一行关键判断）

在 `parse_candidates(json, seq)` 开头做：

- `if (seq != s_req_seq) return;`

语义：只允许“最新那次输入”的响应更新候选栏，旧响应一律丢弃。

### 4) UI 与网络任务解耦

对话里采用的套路是：

- 网络请求跑在独立 FreeRTOS task（HTTP + JSON 解析）
- 解析完只更新：
  - `s_cand[]`
  - `s_cand_ready=true`
- LVGL/UI 线程用 `lv_timer`（例如 100ms）轮询 `s_cand_ready`，再刷新候选栏 UI

要点：**不要在网络 task 里直接操作 LVGL 对象**。

---

## 适用范围（不止输入法）

只要是“用户连续触发 + 请求可能乱序返回”的场景都适用：

- 搜索联想
- 自动补全
- 扫描设备列表（刷新）
- 任何“按键/触摸每次变化就发请求”的 UI

---

## 频率控制（建议）

即使你做了 `seq`，高频请求仍然会带来：

- CPU/网络占用增加
- UI 线程更忙（候选栏频繁刷新）

对话里的建议是：在设备侧做防抖（debounce）：

- 停止输入 300ms 后再发请求

`seq` 解决“乱序覆盖”，debounce 解决“请求太多”。

---

## 验收标准

- 快速输入/删除拼音（连续触发 10+ 次请求）：
  - 候选栏不会倒退到旧结果
  - UI 不闪烁、不出现“突然跳回上一轮候选”
