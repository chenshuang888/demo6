## 一句话结论

**不要让 UI 代码知道“320/240/20/24/296”这种数字。**把它们变成：

- 运行时取值：`lv_display_get_hor_res()` / `lv_display_get_ver_res()`
- 或集中常量：`ui_metrics.h`（状态栏高、导航栏高、padding、可用区域计算）
- 宽度优先用 `lv_pct(100)`，减少“换屏必改文件”的维护成本

---

## 典型触发场景

- 换屏幕方向（横屏↔竖屏）
- 换屏幕分辨率（240×320 ↔ 320×240 / 360×360）
- 需要同一套 UI 适配多块屏（例如不同批次硬件）

如果你的项目里几乎搜不到 `lv_pct`、`lv_display_get_*_res`，但能搜到大量 `320`，基本就属于“全硬编码 UI”。

---

## 改造策略

### 1) 把“屏幕指标”收敛到一个地方

建议新增一个 `ui_metrics`（文件名随项目）：

- `ui_get_hor_res()` / `ui_get_ver_res()`
- `UI_STATUS_BAR_H`
- `UI_NAV_BAR_H`
- `ui_get_content_h()` = `ver_res - UI_STATUS_BAR_H - UI_NAV_BAR_H`
- `ui_apply_full_width(obj)` = `lv_obj_set_width(obj, lv_pct(100))`

目标：上层页面不要自己算“296 从哪来的”。

### 2) 把硬编码宽度替换为“满宽”

优先替换模式：

- `lv_obj_set_size(xxx, 320, h)` → `lv_obj_set_width(xxx, lv_pct(100))`
- `lv_obj_set_size(xxx, 240, h)` → 同上

高度是否要固定，取决于组件语义（状态栏/导航栏通常固定高度；内容区高度通常由父容器或 flex/grid 决定）。

### 3) 触摸坐标映射：能不手写就不手写

对话里强调的关键点是：

- 分辨率宏定义 + 触摸坐标映射逻辑是“最关键改动点”
- 如果你已经回到竖屏，原本为了横屏写的 `x/y` 互换、取反逻辑应该删掉或统一抽象

推荐原则：

1. 优先用驱动/面板层的旋转配置（统一在 HAL 层处理）
2. 真需要手写映射时，把映射写成**清晰的变换函数**，并配套校准/测试步骤（别散落在回调里）

## 快速迁移清单（以“320→240”这类修改为例）

对话中给过一个很实用的做法：先全局搜“320/240”等数字（排除字体文件），列出需要改的文件与行，再逐个收敛成“指标/百分比”。

常见命中位置：

- `main.c`：`LCD_H_RES/LCD_V_RES`、触摸坐标映射
- `status_bar.c`：`lv_obj_set_size(bar, 320, 20)`
- `app_manager.c`：`nav_bar`、返回/主页按钮尺寸
- `launcher.c`：桌面 grid 宽高
- `app_settings.c`：分隔线宽度

**改完一遍以后，要把这些地方继续改造成 `lv_pct(100)` / `lv_display_get_*_res`**，否则下次换屏还要再挨个改。

---

## 验收标准

- 把屏幕从 320×240 改到 240×320（或反过来）：
  - 不需要修改 5+ 个 UI 文件（理想是只改 HAL/metrics）
  - 触摸坐标不需要“猜”出来（有固定的映射/旋转配置）
- UI 关键页面（Launcher/Settings/Clock）：
  - 不出现控件超出屏幕边界
  - 状态栏/导航栏在所有页面都稳定可见
