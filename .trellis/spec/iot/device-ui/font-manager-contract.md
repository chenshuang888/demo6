# Contract：`font_manager` 的职责边界与稳定接口（UI 只依赖 role，不依赖路径）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo5.md`）中对 `font_manager`/`ui_app`/资源目录的系统性梳理：第一版最小 API、哪些后置不做、以及未来接入设备文件系统时哪些接口必须保持稳定。
>
> 目标：保证字体系统能演进（从内置/嵌入式 → 文件系统加载 → 子集/缓存），但 UI 层代码不被路径/格式/目录结构绑死。

> **[本项目适配说明]**（demo6）：
> - 本项目**未独立拆 font_manager**。字体在 `app/app_fonts.c` 的 `app_fonts_init()` 直接用 `lv_tiny_ttf_create_data_ex` 创建，UI 层通过三个全局 `g_app_font_text/title/huge` 引用
> - 原因：只有一份 TTF（中文子集 + Montserrat fallback），三个字号，字体来源单一（EMBED_FILES），不需要"多源加载 + 路径抽象"
> - 本 contract 作为**将来拆 font_manager 时的参考**：如果引入多字体源（文件系统 / OTA 下发 / 多字库切换），UI 层应按本契约的 "依赖 role 而非路径" 原则改造
> - 当前替代实现：`./tiny-ttf-plus-fontawesome-fallback-playbook.md`
> - **复用时**：本契约的不变式（UI 只依赖 role、字体生命周期归 manager、不变接口放首位）通用；具体 API 命名需要按项目实际命名

## 一句话结论

**UI 层只关心“字体角色（role）”，不关心“字体在哪里/怎么加载”；`font_manager` 对外稳定暴露 `get(role)`，内部实现允许迭代。**

## UI 直接按路径加载字体行不行？

对话里的态度很务实：

- **短期、小项目**：UI 直接调用 `lv_binfont_create()` 或自己 `fopen` 按路径加载，确实“完全能跑”，也不一定立刻出事
- **一旦项目变大**：目录/命名/缓存/回退/多页面复用会“悄悄变散”，最后难维护的不是功能，而是资源与调用的耦合

对话里给出的“会散”的典型方式包括：

- 路径字符串散落到 UI 各处（改目录/改命名就得全局搜）
- 同一路径被重复 `create`（重复占内存/重复解析）
- 字体对象生命周期难管（谁持有、何时销毁、是否泄漏）
- fallback 逻辑散到每个页面（每个页面一套 `if (!font) ...`）
- UI 语义退化成“文件名驱动”（UI 讨论的不再是标题/正文，而是某个 `.bin` 文件名）

`font_manager` 的价值就在于：把“字体加载实现细节（路径/格式/来源/缓存/回退）”收敛在一个点，让 UI 只依赖 role，从而允许你后续把实现从“内置字体”无痛替换成“文件系统字体/子集/缓存策略”。

### 如果你坚持 UI 直接访问路径（不推荐但可行）

对话里建议至少守住 4 条底线（否则问题只是被延后爆）：

1. 路径常量集中定义（禁止每个 UI 文件手写字符串）
2. 禁止同一路径在多个地方重复 `create`
3. 生命周期统一（全局一次创建，或页面进入创建/退出销毁，二选一并写清）
4. fallback 统一（不要每个页面各写一套）

---

## `font_manager` 应该负责的事

- 初始化字体系统（一次性：挂钩 LVGL、准备 cache、加载默认字体）
- 基于“字体角色”提供 `lv_font_t*`（例如正文/标题/数字/图标）
- 管理失败策略（缺字/缺文件时 fallback 到默认字体或报错提示）
-（可选）做最小缓存，避免重复解析/创建字体对象

---

## `font_manager` 不应该负责的事（对话中的强建议）

- 不要让 UI 通过路径拿字体（例如 `font_manager_load("/res/fonts/xxx.bin")`）
- 不要过早引入通用 `resource_manager` 把事情做重
- 不要把 `resources/fonts/images/icons` 的目录结构直接绑定到 UI API（目录可以变，UI 不该变）

---

## 建议稳定下来的外部接口（未来接入文件系统也不改）

对话里明确建议“尽量保持稳定”的接口形态是：

- `font_manager_init(void)`：初始化入口（内部可以从内置改成从 `/res` 加载）
- `font_manager_get(font_role_t role)`：按 role 获取字体
- `font_role_t`：role 枚举/定义（UI 依赖它）
- UI 只 include `font_manager.h`（不要 include LVGL font 细节/路径细节）

---

## 内部实现允许变化的地方

对外接口不变的前提下，内部可以逐步演进：

- 字体来源：内置 C 字体 / EMBED_FILES / `/res` 文件系统
- 字体格式：binfont / Tiny TTF（取决于你的路线）
- cache 策略：无 cache → 固定大小 cache → LRU 等
- fallback 策略：缺字统计 → 子集扩充流程

## 字体加载与卸载策略（RAM 够用时先别上重方案）

对话里有一段很关键的澄清：**“按字现取现销毁”比看起来难很多**，原因包括：

- LVGL 会因为刷新/动画/遮挡/换行/样式变化等反复重绘，同一个 glyph 会被反复查询
- 渲染不只要 bitmap，还要 metrics（`glyph_dsc`）、cmap、desc 表、bitmap 区等
- LVGL 对外暴露的是稳定可复用的 `const lv_font_t*`；如果中途销毁，UI 对象可能仍在引用

因此对话里给出的可落地分级是：

### 方案 A：按“字体”懒加载（最现实，改动小）

- 首次用到某 role 时再 `lv_binfont_create()` 加载对应字体
- 优点：立刻减少“启动时全加载”的浪费，兼容现有 binfont 路线
- 缺点：一旦加载，仍然可能是整套字体进 RAM（不是按字粒度）

### 方案 B：按“页面/场景”加载，用完卸载（平衡，但需要管理引用）

- 例如某些页面才需要标题 24px，离开页面后卸载
- 关键约束：卸载时必须确保当前没有任何对象还在引用该 `lv_font_t*`

### 方案 C：按“字符”动态取（极致省 RAM，但工程复杂度很高）

通常意味着：

- 自己实现 `lv_font_t` 的回调（`get_glyph_dsc` / `get_glyph_bitmap`）
- 字体文件格式要支持快速随机访问
- 需要缓存策略（LRU、常用字 cache、页面 cache）

对话里的结论是：在 ESP32 项目里，优先顺序通常是 A → B；C 不适合作为小改动方案。

## 一个容易混淆的点：覆盖范围 vs 运行时常驻

对话里提醒：你说“不要每一个必须得拿出 3500 字”，要分两件事：

- **生成阶段（覆盖范围）**：`ui_sans_16/24` 是否包含 3500 字，是“缺字概率/通用性”的问题
- **运行阶段（RAM 常驻）**：运行时是否必须把整套字体长期留在 RAM，是“加载/缓存/卸载策略”的问题

不要把这两件事混成一个问题一起解。

---

## 第一版目标（最小可落地）

对话里建议第一版只做“少而稳”：

- 两套字体（UI 默认小字体 + 标题字体）
- 一个数字/图标字体（如果确实需要）
- 一个最小 `role -> font` 映射表（哪怕是数组/switch）

其余能力（动态更新、复杂资源系统、自动裁剪流水线）后置。

---

## 验收标准

- UI 层从不出现资源路径（`/res/...`）或字体文件名
- 更换字体来源（内置 → 文件系统）时，UI 不改，只改 `font_manager` 内部
- `font_manager` 的边界清晰：它负责“给我一个可用字体”，不负责“UI 如何排版/页面如何组织”
