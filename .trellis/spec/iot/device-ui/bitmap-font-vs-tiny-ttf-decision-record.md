# Decision Record：位图字体（binfont/内置）vs Tiny TTF（动态渲染）怎么选

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo6.md`）中围绕“中文字体占空间/占 RAM、为什么最终选 Tiny TTF、glyph cache 为什么 256 足够、以及 `APP_FONT_*` 宏为什么能无痛切换实现”的系统性解释。
>
> 目标：把“字体路线选择”从情绪化（看起来更高级/更省）变成工程化（资源账本 + 可回退 + 可验收）。

---

## 结论（适用于本项目形态）

- 想要“字库覆盖广 + 不想频繁重生成字库”：优先 Tiny TTF
- 想要“极致性能 + 字符集很小且稳定”：位图字体更省心

对话在 demo6 的最终收敛是：**Tiny TTF + 子集 TTF + glyph cache**（并配合分区扩容）。

---

## 根本区别：字形用什么方式记录

- 位图字体：字形在“生成阶段”就被栅格化成 bitmap，固件运行时只做查表拷贝
- Tiny TTF：字形以矢量轮廓存于 TTF（flash），运行时按需把轮廓栅格化成 bitmap，并缓存常用 glyph

---

## 资源账本（工程视角）

| 维度 | 位图字体 | Tiny TTF |
|---|---|---|
| Flash | 随“字数×字号×bpp”线性涨 | 主要由字体源的 `glyf` 决定；同字数下黑体更紧凑 |
| RAM | 可接近 0（若字体在 flash）或较大（若整库进 RAM） | cache 占用可控（如 256 glyph），其余在 flash |
| 首次渲染延迟 | 基本无 | 可能有 1–5ms（cache miss 时栅格化） |
| 缺字处理 | 缺字必须重生成 | 缺字可通过扩子集 TTF 迭代（流程化） |

## “开关”怎么设计：让 UI 不感知底层实现

对话里强调：上层页面不要到处写“具体字体变量名”，统一走 `APP_FONT_TEXT/TITLE/LARGE` 这类 role 宏。

切换两种实现时，UI 侧只需要满足：

- 最终拿到的是 `lv_font_t*`

唯一的小差异（对话点名）：

- 位图/内置字体常是 `const lv_font_t`：宏可能写成 `(&lv_font_xxx)`
- Tiny TTF 返回堆对象指针：宏直接写成 `(g_font_ptr)`

因此推荐的工程做法是：

- 统一在 `app_fonts.c/.h` 或 `font_manager` 内封装角色宏
- 页面只 include 角色头文件，不碰字体实现细节

## cache 直觉校准（为什么 256 常常够）

对话里的核心直觉：

- 屏幕上同一时间显示的字很少（几十到几百），远小于字库总字数
- LRU cache 覆盖“近期常用字”即可获得高命中率

因此默认 `cache_cnt=256` 往往是“足够好”的起点，真不够再升 512/1024。
