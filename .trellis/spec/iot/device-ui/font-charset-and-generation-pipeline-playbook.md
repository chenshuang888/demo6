## 一句话结论

不要把字符集藏在脚本里、也不要靠 Unicode 范围猜；把“3500 字”本身入库成**单一真源资产**，再用一个固定脚本生成 4 个 binfont 产物并入库，最后用统一的验证格式记录“真实占用 + 设备显示效果”。

---

## 1) 字符集资产：单一真源（Single Source of Truth）

对话里强烈不建议：

- 用 Unicode 范围（会超出很多不需要的字）
- 在脚本里随手拼字符串（不可审查、不可复现）
- 从页面文案反推字符集（一定会漏字）

推荐做法：

- 把“3500 常用字”定义为仓库内的正式资产文件（例如纯文本 `charset_3500.txt`）
- 以后任何字体生成都只读这个文件

---

## 2) 目录结构：工具与产物分离

对话里建议把“生成工具链”与“运行时资源目录”分离：

- 工具目录（TTF 源文件、字符集、脚本）不放在 `resources/`
- 生成出来的 `.bin` 才落到 `resources/fonts/`

一种推荐形态（示例）：

```text
tools/fonts/
  source/        # TTF/OTF 字体源（正式字体源也放这里）
  charset/       # charset_3500.txt / charset_ui_title_subset.txt 等
  scripts/       # 生成脚本（固定入口）
resources/fonts/
  ui_sans_16.bin
  ui_sans_24.bin
  ui_num_24.bin
  ui_icon_16.bin
```

---

## 3) 产物策略：按 role 拆分，避免无意义膨胀

对话里建议保持 4 个稳定产物名（不改 UI 调用方式），只把“中文主字体资产”正式化：

- `ui_sans_16.bin`：承载“全量 3500 字”的正文中文（建议偏清晰）
- `ui_sans_24.bin`：标题中文（可以全量，也可以子集；看资源预算）
- `ui_num_24.bin`：只保留数字/时间/少量符号子集
- `ui_icon_16.bin`：只保留图标子集（后续替换为真正 icon 字库）

关键点：

- “完整 3500”最多扩到 1~2 份主中文字体，不要让 `num/icon` 也变成大字库

---

## 4) 流程固化：一个脚本入口 + 生成结果入库

对话里推荐的“不过度复杂”的固化方式是：

- 一个工具目录
- 一个脚本入口（固定参数/固定产物路径）
- 生成结果 `.bin` 入库（避免“每个人本机生成不一致”）

第一阶段可先固定为“验证链路稳”为目标（例如先用 `--no-compress --no-prefilter`），等稳定后再优化压缩/体积。

---

## 5) 验证方法：把“真实占用”与“显示效果”都验证到位

对话强调：字体这类资源容易出现“看起来跑了，但其实没生效”的假象，原因之一是 fallback。

### A. 先明确你要看哪一种“flash 占用”

至少区分：

- `.bin` 文件大小（主机侧）
- `resources` 分区镜像大小/占用（构建侧）
- 设备侧实际效果（是否真的加载成功，而不是 fallback）

### B. 设备侧验证（必须）

建议至少做到：

- 日志明确显示 `loaded font: /res/fonts/...`（不是 `use fallback...`）
- 页面上能看出正文/标题/数字字体确实按 role 生效

### C. 输出结论格式（建议统一）

对话里推荐你最终能产出一条“可复现结论”：

- 使用的字体源（TTF 路径/版本）
- 使用的字符集资产版本（charset 文件 hash/版本）
- 生成参数（bpp/是否压缩/是否预过滤）
- 4 个产物大小
- `resources` 分区总预算与剩余空间
- 设备端日志与截图/观感结论

---

## 风险提醒（对话中的两个重点）

1. **仓库状态与已验证状态不一致**：会导致你测出来的占用与实际不可复现  
2. **fallback 会掩盖真实收益**：看起来 UI 没崩，但其实新字体没加载成功
