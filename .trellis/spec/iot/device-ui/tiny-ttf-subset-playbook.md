## 前置条件

- ESP-IDF + LVGL（能跑 demo）

## 关键约束（踩过的坑先写出来）

### 约束 1：必须是 `.ttf`（TrueType）

对话里明确提醒：Tiny TTF 底层常基于 `stb_truetype`，**只解析 TrueType outlines**。

- ✅ `.ttf`：可用
- ⚠️ `.otf`（CFF）：可能不可用（除非你确定是 TrueType OTF 或已转换）

### 约束 2：不追求“全字库”，追求“够用”

对话中的实操策略是：

- 先用 GB2312 常用字 + ASCII + 符号（再按缺字增量扩充）
- 不要一开始就做全量 CJK（体积与验证成本都暴涨）

---

## 关键经验：子集 TTF 大小主要由字体“字形轮廓数据（glyf）”决定

对话里对 3.4MB 子集字体做了 table 账本拆解，结论非常硬核：

- `glyf` 往往占 **95%+**
- 其他表（`cmap/hmtx/loca/GPOS/kern/...`）加起来可能只有几十 KB

因此：

- 想靠“再 drop 几张表”把 3.4MB 砍到 1.5MB，通常**不现实**
- 真正有效的杠杆是：**换字体源（黑体更紧凑）**，或减少字符集规模

对话中的典型对比：

- 楷体（曲线节点多）：同字数下每字节数更大
- 黑体（笔画平直）：同字数下更容易压到 1.2–1.8MB

---

## “不减字数还要更小”的现实解法：换更紧凑的黑体源

对话里实际把“楷体子集 3.4MB”切换到“黑体子集 ~1.57MB”，字数保持不变，靠的就是字体源的差异。

工程化建议：

- 如果你有“硬性大小线”（例如 1.5MB），先把它当作一个目标值，但不要迷信：
  - 当 app 分区是 6MB 时，1.57MB vs 1.50MB 对实际系统没有本质影响
  - 更该关注的是“固件整体占用占分区比例”与“渲染性能/缺字率”

## 落地步骤（推荐顺序）

### 步骤 1：选字体源

对话中的经验：**小屏更偏黑体**，并且在相同字符数下更容易压缩；楷体曲线多，子集后也可能偏大。

### 步骤 2：用 `pyftsubset` 做子集

目标字符集建议包含：

- GB2312 常用字（如果你 UI 以常用中文为主）
- ASCII（英文/数字/符号）
- 你 UI 真会显示的“状态字/地名/歌名等”
- LVGL 符号（LV_SYMBOL 区，避免图标缺失）

输出：`xxx_subset.ttf`

### 步骤 3：把子集字体嵌入固件（CMake）

对话中的落地方式：

- 在 `app/CMakeLists.txt`（或你的组件 CMake）中加入：
  - `EMBED_FILES "fonts/xxx_subset.ttf"`

### 步骤 4：在 `app_fonts.c` 创建 Tiny TTF 字体对象

对话里点名的 API：

- `lv_tiny_ttf_create_data_ex(data, size, size_px, kerning, cache_cnt)` → 返回 `lv_font_t*`

关键点：

- 保持你的宏名不变（如 `APP_FONT_TEXT/TITLE/LARGE`）
- 让宏最终都指向 `lv_font_t*`
- 这样所有页面代码“零改动”

### 步骤 5：sdkconfig 配置

对话中的关键开关：

- 开：`LV_USE_TINY_TTF=y`
- 结合情况关闭内置大 CJK 字库，回收 Flash（对话里提到量级可达 ~800KB）
- glyph cache：对话中使用了 `cache_cnt=256` 作为工程可用的起步值

### 步骤 6：必要时调整分区（避免“卡死在 4MB 红线”）

对话中提到的现实问题：

- 子集字体可能从 1.5MB 漂到 3.4MB（取决于字体源与字符集）
- 如果 app 分区太紧，直接扩容更省事（例如 4MB → 6MB），storage 相应缩小

> 原则：不要为了省一点分区空间而让字体/功能方案变复杂（这是典型“过度优化导致过度设计”）。

## glyph cache（`cache_cnt`）怎么选：为什么 256 通常够用

对话里给出的判断依据是：

- UI 任意瞬间在屏幕上显示的字数，远小于字库总字数（几千/几万）
- cache 用 LRU 淘汰策略，命中的是“近期在屏幕上反复出现”的字（标题、菜单项、常用词）

因此起步值推荐：

- `cache_cnt=256`：多数 UI 足够（对话称“绰绰有余”的起点）
- 如果你发现频繁 miss（肉眼可见卡顿/日志表明反复解析同一批字），再升到 512/1024

注意：cache 变大是“空间换时间”，但真正的代价往往不是运行时，而是**你要如何验证/定位 cache 行为**。先从 256 起步更符合奥卡姆剃刀。

## 验证要点（对话里给的很实用）

- 启动日志：确认 PSRAM、分区占用比例等关键信息正常
- UI 验证：挑“历史缺字”的场景（歌名/通知/地名/常用短语）逐个验证
- cache 行为：
  - 首次渲染某字可能有 1–5ms 延迟
  - 后续命中 cache 应明显更快
- LV_SYMBOL 图标：应走 fallback 机制，不因换字体而丢图标

---

## 风险与回退

- 如果 build 超出分区：优先扩分区；或换更紧凑的字体源重新子集化
- 如果缺字：扩充字符集并重新子集化（保持流程不变）
- 如果 UI 卡顿明显：调 cache、优化渲染频率，必要时回到“位图字体 + 小字符集”
