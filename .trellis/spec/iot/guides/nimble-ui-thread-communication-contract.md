## 一句话结论

- **BLE 回调（NimBLE host task）只做两件事：解析 + 非阻塞投递（queue）。**
- **系统时间/业务快照的最终写入只在 UI 线程发生。**
- “连接状态”这类 **单写多读 + 标量** 可用 `volatile`（仅用于禁止编译器缓存），复杂数据必须用队列或锁。

---

## 线程模型（对话中的抽象）

- `ui_task`：你的 UI 主循环（每 10ms 刷新、执行 `page_router_update()`、`lvgl_port_task_handler()`）
- NimBLE host task：BLE 协议栈事件循环（GAP/GATT 回调在这里触发）

我们关心的约束是：

- BLE 回调不能阻塞（否则影响协议栈）
- LVGL 对象树只能在 UI 线程修改（否则崩/竞态）

## 推荐模式：队列化请求 + UI 线程应用

### 场景 A：同步时间（CTS 写入）

对话收敛的最佳实践：

1. BLE 回调收到 CTS 写入 → **把“要设置的时间”投递进队列**（非阻塞）
2. UI 线程每帧调用 `time_manager_process_pending()`：
   - 从队列取出所有待处理项
   - **在 UI 线程中调用 `settimeofday()`**（成为系统时间唯一写入点）
3. UI 页面不需要知道 BLE 的存在：
   - 页面 `update()` 里读系统时间（`time()`/`localtime`），发现变化就重绘

关键约束：

- `xQueueSend` 必须是 non-blocking（队列满则返回错误，让 BLE 客户端重试；不要静默丢）
- BLE 回调里不直接 `settimeofday`（避免与 UI 按钮调整时间的 RMW 竞态）

### 场景 B：天气/音乐等数据推送

同构模式（对话称为“完全同构”）：

- BLE 回调只投递 payload
- UI 线程消费队列并更新“最新快照”
- 页面 `update()` 里读取快照并渲染

## `volatile` 的适用边界（对话的决策框架）

`volatile` 只解决一件事：**阻止编译器把变量缓存到寄存器，强制每次从内存读写**。

### 什么时候 `volatile` 足够

- 单写多读
- 类型是“单次访问原子”的标量（如 `bool`、对齐的 `uint32_t` 指针）
- 不做 RMW（不做 `x++` 这种读改写）

典型例子：`static volatile bool s_is_connected;`

### 什么时候必须队列/锁

- `struct`/数组/64 位类型（可能撕裂）
- 多写者
- RMW（读-改-写）逻辑（例如系统时间调整）

---

## 验收标准

- BLE 回调不阻塞（写入请求立即返回）
- UI 线程是系统时间唯一写入点（BLE 写时间与 UI 按钮调整不再互相打架）
- 连接状态等标志位 UI 能稳定看到变化（不被编译器缓存成“永远 Off”）
