## 一句话结论

- **UI 线程 ≠ 触摸回调。**
- **所有 LVGL 对象树的修改，只能在 UI 线程执行。**
- **回调里只做“翻译/发消息”，耗时 IO 交给后台 logic/task，结果再回 UI 更新。**

---

## 规则 1：LVGL 对象树的所有改动 = 只能在 UI 线程

典型改动包括：

- 创建/删除/修改 `lv_obj_t`
- 改样式、改文本、改图片、切屏
- 任何会触发 LVGL 内部状态变化的操作

---

## 规则 2：触摸/按钮回调允许做什么、不允许做什么

### 回调里允许（推荐）

- 组一个“请求消息”（command），投递到 logic/后台队列
- 更新少量本地状态（不触发 IO）
- 启动/停止一个 timer（由 UI 线程处理）

### 回调里不建议（会导致失控）

- 网络请求（HTTP/MQTT/下载）
- 读写存储（文件系统/NVS/数据库）
- 大量计算/循环（阻塞 UI）
- 在回调里串起“业务流程大闭环”

---

## 规则 3：是否需要后台线程，用“会不会阻塞/会不会重计算”来判断

不要用“这是不是逻辑”来判断要不要开线程。更实用的判断：

- 会阻塞（网络/存储/等待外设）→ 放后台
- 会重计算（长循环/大解析）→ 放后台
- 只是更新 UI 状态（轻量）→ 留在 UI 线程

---

## 最小可落地的消息模式（UI → logic → UI）

1. UI 回调只发 `UI2LOGIC_CMD`（不阻塞 UI）
2. logic 做耗时事（网络/存储/传感器）
3. logic 发 `LOGIC2UI_MSG` 回 UI
4. UI 的 `ui_on_msg()` 更新 label/图标/状态

验收点：

- UI 点击“立即响应”（哪怕数据稍后更新）
- 网络慢/存储慢不会卡住 UI
