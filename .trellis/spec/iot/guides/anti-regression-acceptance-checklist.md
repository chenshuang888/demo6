# 防复发验收清单（IoT/嵌入式版）

> 目标：每次新增条目或做一次重构后，都能回答——“下一次遇到同类场景，会不会还踩同一个坑？”

---

## 一、条目质量（写得对不对）

针对每个新蒸馏条目（Playbook/Pitfall/Contract/Decision）检查：

- [ ] **有一句话结论**：能让人 10 秒内知道“该怎么做”
- [ ] **能定位到触发条件**：在什么场景下适用/不适用
- [ ] **能验收**：有可执行的验证顺序或验收标准（不是“应该没问题”）
- [ ] **不编造细节**：只固化对话中已经形成稳定结论/流程的内容

---

## 二、跨层一致性（最容易复发的坑）

当条目涉及 2 层以上（设备↔主机工具↔Web/云）时：

- [ ] **契约单一事实来源**：协议/字段语义/状态机转移写在 `Contract`，不要散落在多个文档
- [ ] **命令表/错误码一致**：MCU 与主机工具必须同步更新（新增命令不能只改一端）
- [ ] **单位与语义统一**：例如时间单位、大小单位、slot/status 的含义

---

## 三、并发与持久化（固件侧高危）

涉及 NVS/flash/配置落盘时：

- [ ] 是否遵守 **single-writer**：只有一个任务写入，其他线程只投递请求
- [ ] 是否明确了“写入点”与“执行顺序”（避免隐式并发）
- [ ] 是否给出性能演进路径（例如需要时拆 `persist_task`，但语义不变）

---

## 四、主机端工具（页面/脚本高危）

涉及 PC/Web/Electron 工具链时：

- [ ] 共享脚本是否能在任意页面加载而不崩（DOM 引用判空/延迟绑定）
- [ ] 日志/提示是否具备 fallback（不因为缺 DOM 元素而让页面整体挂掉）

---

## 五、场景回放验收（真正检验“不会再错”）

建议维护一个“回放用例列表”（你可以逐步扩充到 10~20 个）：

1. LVGL 接入：显示 OK，但触摸坐标反了 → 能否迅速定位到“坐标映射只放 lvgl_port”
2. 增加新页面：引入共享脚本 → 能否避免 DOM 依赖导致加载即崩
3. BLE 推送频繁 + UI 渲染：偶发配置错乱 → 能否命中 single-writer/NVS 竞态条目
4. OTA：分包传输偶发卡住 → 能否检查 ACK(seq) 与状态机状态

验收通过标准（建议）：

- AI 仅依赖 `spec/` 即可给出：**定位路径 + 根因候选排序 + 最小修复 + 预防清单**
