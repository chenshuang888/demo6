# Playbook：从代码库证据驱动地写 README（避免“拍脑袋架构”）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-----.md`）中“遍历大量项目 → 产出技术栈清单/项目分析报告/README 草案”的工作流。
>
> 目标：把“能稳定产出靠谱 README”的方法固化下来，让 AI/人写 README 时**不编造**、不漏关键约束，并能快速把架构主线讲清楚。

---
## 上下文签名（Context Signature，必填）

> 目的：避免“场景相似但背景不同”时照搬实现细节。

- 目标平台：ESP32/ESP32‑S3/STM32/…（按项目填写）
- SDK/版本：ESP-IDF x.y / LVGL 9.x / STM32 HAL / …（按项目填写）
- 外设/链路：UART/BLE/Wi‑Fi/TCP/UDP/屏幕/触摸/外置 Flash/…（按项目填写）
- 资源约束：Flash/RAM/PSRAM，是否允许双缓冲/大缓存（按项目填写）
- 并发模型：回调在哪个线程/任务触发；谁是 owner（UI/协议/存储）（按项目填写）

---

## 不变式（可直接复用）

- 先跑通最小闭环（冒烟）再叠功能/优化，避免不可定位
- 只直接复用“原则与边界”，实现细节必须参数化
- 必须可观测：日志/计数/错误码/抓包等至少一种证据链

---

## 参数清单（必须由当前项目提供/确认）

> 关键参数缺失时必须停手，先做 Fit Check：`spec/iot/guides/spec-reuse-safety-playbook.md`

- 常量/边界：magic、分辨率、slot 大小、最大 payload、buffer 大小等（按项目填写）
- 时序策略：超时/重试/节奏/窗口/幂等（按项目填写）
- 存储语义：写入位置、校验策略、激活/回滚策略（如适用）（按项目填写）

---

## 可替换点（示例映射，不得照搬）

- 本文若出现文件名/目录名/参数示例值：一律视为“示例”，必须先做角色映射再落地
- 角色映射建议：transport/codec/protocol/ui/persist 的 owner 边界先明确

---

## 停手规则（Stop Rules）

- 上下文签名关键项不匹配（平台/版本/外设/资源/并发模型）时，禁止照搬实施步骤
- 关键参数缺失（端序/magic/分辨率/slot/payload_size/超时重试等）时，禁止给最终实现
- 缺少可执行的最小冒烟闭环（无法验收）时，禁止继续叠功能

---


## 目标

- 在不了解项目的情况下，快速形成“可验证的项目画像”
- 产出 README 的最小闭环：能跑、能复现、能定位问题
- 架构描述来自**证据**（代码/配置/目录结构），不来自猜测

---

## 设计边界（避免过度设计/过度承诺）

- 不写“理想架构”，写“当前真实架构”
- 不清楚的地方允许写“未知 + 如何确认”（比瞎写强）
- 对 IoT/嵌入式项目，README 必须包含：硬件/外设/采样率/波特率/协议/时序等**硬约束**

---

## 实施步骤（建议顺序）

### 1）盘点项目类型与入口（先确定“读什么”）

按证据决定项目类型（嵌入式/前端/后端/脚本工具/混合），并明确入口文件：

- 嵌入式：`main.c/main.cpp`、`app_main()`、`startup`、`CMakeLists.txt`、`*.ioc`、`Keil/IAR` 工程文件
- 后端：`server.js`/`app.py`/`main.go`、路由入口、配置加载位置
- 前端：`package.json`、`vite.config.*`、`src/main.*`、路由与状态管理入口
- 工具链：`scripts/`、`requirements.txt`、CLI 参数解析点

输出物：一张“入口点列表”（带文件路径），不要只写“用到了 XXX 技术栈”。

---

### 2）建立目录证据（不要一上来写结论）

最少要抓到三类证据：

1. 目录树（排除大文件夹如 `.git/`、`node_modules/`、`build/`）
2. 关键配置文件（构建/依赖/环境变量）
3. 关键模块文件列表（按“职责”分组，而不是按目录硬贴）

---

### 3）抽“架构主线”（数据流/控制流优先）

建议用 3 条线索抽主线：

- **数据流**：数据从哪来 → 怎么处理 → 到哪去（例如采集→处理→识别→对话→播放）
- **控制流/状态机**：谁触发谁、状态怎么推进、失败怎么回滚
- **跨层契约**：协议/Schema/版本兼容/错误码/超时重试（IoT 里是核心）

输出物：README 中的一段“从 0 到 1 的链路说明”，必须能对应到代码模块。

---

### 4）写“模块职责表”（把可维护性写出来）

把关键文件/模块按职责列成表（示例字段）：

- 模块名
- 职责（1 句话）
- 输入/输出（数据结构或协议名）
- 关键约束（采样率/波特率/内存上限/线程模型）
- 常见故障（如何观测/下一步查哪里）

---

### 5）把“运行/复现”写成可执行步骤（别写口号）

README 的运行步骤必须满足：

- 可复制（命令可直接运行）
- 有前置条件（驱动/串口号/依赖安装）
- 有验证点（成功时看到什么输出，失败时查哪里）

对于嵌入式项目，必须拆成：

1) 主机侧（脚本/工具）怎么跑  
2) 设备侧（编译/烧录/串口 monitor）怎么跑  
3) 两端协议/参数如何对齐（例如波特率、采样率、帧格式）

---

### 6）列“未知项清单”（这是防编造的关键）

把你没有证据确认的内容显式列出来：

- 未确认的硬件型号/引脚
- 未确认的协议版本/字段含义
- 未确认的构建方式（Keil/IDF/PlatformIO）

并附上“如何确认”的最短路径（要查的文件/命令/日志位置）。

---

## 验证顺序（推荐）

1. 最小冒烟：只跑通最短链路（例如只跑 host tool 的 `--dry-run`）
2. 功能验证：按 README 步骤跑一次，确保没有跳步
3. 边界验证：挑 1～2 个最常见的失败点，README 给出明确排查入口

---

## 常见问题（快速排障）

- README 写得“像真的”，但没人能跑：通常是缺少前置条件/验证点/版本约束
- 架构讲得很漂亮，但代码对不上：通常是先写了结论，没做“入口证据→主线→模块职责”的链路
- IoT 项目最常漏：波特率/采样率/协议字段/超时策略/设备状态机
