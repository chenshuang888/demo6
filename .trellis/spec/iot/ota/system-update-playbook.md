# Playbook：系统更新（OTA）落地顺序（先基础设施，后页面）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-OLED-----.md`）中“实现顺序建议”。
>
> 目的：避免一上来就写复杂页面，先把协议与回调基础设施做稳，再逐步联调。

---

## 上下文签名（Context Signature，必填）

- 参与方：设备端（MCU/固件）+ 主机端工具（PC/Web/Electron/Node 等）
- 协议形态：通常是“分包 + ACK(seq) + 状态机”的串口/链路协议（契约建议以 `spec/iot/protocol/downloader-protocol-contract.md` 为单一事实来源）
- 风险等级：高（涉及写 Flash/切换版本/重启/回滚）
- 并发模型：必须明确“谁拥有下载/写入状态机”（owner），以及 UI 回调在哪个线程触发

---

## 不变式（可直接复用）

- **先跑通最小闭环再做 UI**：版本检查 → OTA_START/DATA/END → SetActive/重启（或等价动作）
- **协议层与页面层解耦**：页面只消费“进度/结果回调”，不直接读写链路细节
- **可观测性优先**：每一步都要有日志/状态能验收（禁止 silent failure）

---

## 参数清单（必须由当前项目提供/确认）

> 缺任何关键项都应先停手，做 Fit Check/冒烟验证（见 `spec/iot/guides/spec-reuse-safety-playbook.md`）。

- 链路：UART/BLE/Wi‑Fi？波特率/MTU/超时/重试策略是什么
- 分包：chunk 大小、ACK 是否带 `seq`、窗口/重传策略是什么
- 存储：写入位置（内置/外置 Flash）、槽位结构（slot size/slot count）、校验策略（CRC/Hash）是什么
- 激活：SetActive 的语义、是否支持回滚、失败路径如何定义
- UI：页面框架/生命周期、回调线程与 owner 边界（UI 不阻塞、写 Flash 不在 UI 回调）

---

## 可替换点（示例映射，不得照搬）

本 Playbook 中出现的文件名（如 `downloader.h/.c`、`page_settings.c`、`uploader_web.html`）来自某个具体工程，仅用于说明“先底层后 UI”的顺序。

落地到你的工程时，必须先做角色映射：

- **协议/下载状态机 owner**：你的工程哪个模块持有（不要分散在多处）
- **存储写入 owner**：哪个任务/模块负责写 Flash
- **UI 页面对接点**：页面如何订阅进度/结果回调
- **主机端工具**：前端/后端/脚本哪个层负责分包、重试、进度展示

---

## 停手规则（Stop Rules）

- 你还没明确协议契约（命令表/ACK(seq)/状态机）就开始写 UI
- 你还没明确写入位置/槽位/校验/激活语义就开始写“升级成功/失败”的页面逻辑
- 你无法回答“谁是 owner、回调在哪个线程触发”就开始做跨线程联调

---

## 目标

- OTA 版本检查 + 更新流程可跑通
- 进度可回调、可展示、可验收
- 结构清晰：协议层/存储层/页面层职责分离

---

## 设计边界（避免过度设计）

- 先把最小闭环跑通：版本检查 + OTA_START/DATA/END + SetActive
- 先按既有状态机实现；必要优化（例如更细的错误码、更复杂的恢复策略）后置

---

## 实施步骤（建议顺序）

对话中建议的顺序（按“先底层、再 UI、再联调”的思路）：

1. **先完成“协议/回调基础设施”**：添加新命令、回调、状态机（你的工程里可能叫 downloader/protocol/uploader/service 等，按角色映射调整）
2. **再完成“页面框架冒烟”**：新增一个与协议无关的简单页面（用于确认 UI 生命周期与输入链路正常）
3. **再接入“System Update 页面”**：页面只消费进度/结果回调，不直接操作协议细节
4. **整理入口与导航**：把升级入口放到稳定的菜单/launcher 中，避免入口散落
5. **主机端工具补齐协议支持**：确保命令表/分包/ACK(seq)/错误路径与契约一致
6. **按验证顺序联调**：先 UI 冒烟 → 再协议冒烟 → 再 OTA 冒烟 → 最后 UI 联动

---

## 验证顺序（必须可执行）

1. UI 冒烟：About 页能稳定显示（确认“页面系统”没问题）
2. 协议冒烟：版本检查请求/响应能跑通（日志可观测）
3. OTA 冒烟：OTA_START→DATA→END 基本链路走通（ACK 与序列号正确）
4. UI 联动：System Update 页展示进度与最终结果（回调触发正确）
