## 目标

- 用“页面状态机”管理多页面 UI
- 在静止时停止状态机/刷新，降低功耗
- 提供可扩展的页面类型（Title/List/Wave/Modal 等）与统一方法表

---

## 架构总览（核心对象）

WouoUI 的关键是一个全局 UI 对象（示意字段）：

- 刷新控制：`is_motionless`（静止停机）、`slide_is_finish`（滚动动画结束）
- 页面管理：`home_page/current_page/in_page` + `state`（进入/处理/退出）
- 绘图系统：`Pen`、`ScreenBuff`、`Canvas`、`pfun_sendbuff`（刷屏函数指针）
- 动画与效果：`Indicator`、`ScrollBar`、`UIBlur`
- 系统参数：`UiPara`（动画速度/是否循环/滚动步进等参数数组）
- 输入消息：`InputMsgQue`（环形缓冲的消息队列）
- 时间尺度：`time`（ms）
- 页面类型共享：不同页面类型的 `Var` + `Methods`（类似“类 + 虚函数表”的组织）

这套结构的价值在于：**页面只是数据与回调，框架统一调度与渲染**。

---

## 刷新/低功耗策略（非常值得复用）

对话中明确的策略：

- UI 处于静止状态时：停止状态机（不再刷屏、不再做动画步进）
- 只有当发生“动画/输入/状态变化”时才重新驱动刷新

落地建议：

- 把“是否需要刷新”的判定统一收敛到框架层（不要让每个页面自己决定）
- 让动画系统成为“刷新触发器”（有动画 → 需要刷新；动画结束 → 进入 motionless）

---

## 页面管理：状态机 + 过渡页

建议抽象为 3 个阶段：

1. Enter：页面进入（初始化/动效准备）
2. Process：页面运行（按键、消息、动画推进）
3. Exit：页面退出（资源释放/状态保存）

对话中还出现了 `in_page`（过渡中的目标页），可用于：

- 支持切页动画（当前页退出动画 → 进入目标页）
- 避免“切页瞬间把当前页直接销毁导致花屏/状态错乱”

---

## 参数系统（UiPara）：把“体验参数”变成可调资产

对话中 UiPara 的特点是“参数数组化”：

- 动画速度（多种动画分别可调，单位 ms）
- 是否从头展开/是否循环
- 文本滚动步进/模式/延迟等

价值：

- 体验调参不需要改业务代码
- 参数能做“统一默认值 + 页面局部覆盖”

---

## 输入与消息：小容量队列也能稳定

对话中提到 `InputMsgQue` 的消息队列（环形缓冲）：

- 容量很小（示例为 4 条）
- 但语义明确（输入事件/页面消息）

建议：

- 明确队列的“丢弃策略”（满了丢新还是丢旧），并在文档里写死
- 消息类型要少而稳定（不要为了“看起来完整”而定义一堆永远不用的消息）

---

## 验收标准（最小可执行）

1. 多页面切换稳定：不会花屏/卡死/状态错乱
2. 静止不刷新：CPU 占用显著下降（或 tick 触发显著减少）
3. 动画触发刷新：动画期间稳定刷新，结束后自动回到 motionless
4. 新增页面类型可扩展：只新增页面文件与方法表，不改框架核心
