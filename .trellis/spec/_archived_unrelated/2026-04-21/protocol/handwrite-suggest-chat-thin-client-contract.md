## 总体原则

- **ESP32 只负责：输入采集、HTTP 收发、文本显示、候选点选 UI**
- **主机负责：手写识别/拼音/联想、AI API、历史存储**
- 消息尽量“极简 JSON”：字段少、可扩展、易 debug

## 消息类型一览

> 说明：下列 JSON 只描述语义；实际承载可以是 HTTP POST + JSON body。

### 1) 手写识别（recognize）

ESP32 → 主机

```json
{"type":"recognize","seq":1,"strokes":[[{"x":10,"y":5},{"x":30,"y":40}],[{"x":12,"y":18}]]}
```

主机 → ESP32

```json
{"seq":1,"candidates":["你","尔","呢"]}
```

约束：
- `strokes` 为二维数组：每个 stroke 是点序列
- 点坐标使用设备坐标系（主机不做二次“猜测旋转”，旋转应在设备 HAL 层统一）

### 2) 选字联想（select / suggest）

ESP32 → 主机

```json
{"type":"select","seq":7,"char":"你"}
```

主机 → ESP32

```json
{"seq":7,"candidates":["好","们","的"]}
```

建议扩展（可选）：

- 增加上下文，让联想更准：

```json
{"type":"select","seq":7,"char":"你","context":"你"}
```

> 取舍：如果联想依赖 AI 且延迟明显，可以先把联想功能关闭（设备端不再发送 `select`/`suggest` 请求），只保留拼音/手写候选，优先保证输入“不卡”。

### 3) 发起聊天（chat）

ESP32 → 主机

```json
{"type":"chat","seq":42,"message":"你好今天天气怎么样"}
```

主机 → ESP32（流式文本）

- 建议：按纯文本分片返回（chunked/stream），ESP32 逐段 append 到 label/文本框
- 设备端原则：不要求一次收全（避免小 buf 截断）

## 交互流（推荐）

### 输入阶段（手写 + 候选 + 联想）

1. 本地手写区实时画线（0ms）
2. 抬手/停笔后上送 `recognize`，主机回 `candidates`
3. 用户点选某字：
   - 选中字进入输入框
   - 同步发 `select`，主机回“联想候选”
4. 用户可以：
   - 继续写下一个字（再识别）
   - 或直接点联想候选补全短句

### 发送阶段（整句 POST + 输出流式）

1. ESP32 发送 `chat`（整句）
2. 主机调用 AI API，并把回复流式转发给 ESP32
3. ESP32 边收边显示（打字机效果）

---

## 兼容性与演进建议

- `type` 字段作为一级路由；新能力新增 type，不要复用旧 type 的语义
- 主机返回的 `candidates` 统一为数组，长度可变（设备按 UI 容量裁剪显示）
- **建议所有“高频短请求”都带 `seq`**（请求序列号），主机原样回传；设备端只接收最新 `seq` 的响应，丢弃乱序/过期结果（防止候选栏倒退）。
- 流式输出建议支持：
  - “分片文本”
  - “结束标记”（例如连接结束或显式 `{"type":"done"}`，按实现选择）

---

## 验收标准

- 在局域网条件下：
  - recognize→candidates 的往返延迟体感 < 200ms
  - chat 流式回复能稳定逐段显示，不出现“只因 buffer 截断导致 JSON/文本损坏”的白屏类问题
