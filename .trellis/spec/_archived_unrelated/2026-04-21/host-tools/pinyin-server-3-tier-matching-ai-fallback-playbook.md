## 一句话结论

`/pinyin` 用三层策略：

1. **精确字典秒回**（PINYIN_MAP）
2. **音节切分组合秒回**（`split_pinyin` + `combine_candidates`）
3. **AI 兜底**（缩写/无结果，~500ms 级别）

---

## 为什么要三层

- 小屏输入会产生高频短请求（每敲一个字母就请求一次）
- 如果每次都走 AI：
  - 延迟高
  - 不稳定
  - 成本不可控
- 但只靠字典又覆盖不了缩写（例如 `wm`）或异常输入

所以：**80% 走规则，20% 走 AI。**

---

## 三层策略细节

### 第 1 层：PINYIN_MAP 精确匹配（最快）

输入 `"ni"`：

- 直接查表：`["你","泥","逆","拟"]`（秒回）

### 第 2 层：音节切分 + 组合（覆盖多音节）

输入 `"women"`：

- `split_pinyin("women")` → `["wo","men"]`
- 再组合候选：`["我们", ...]`（秒回）

切分策略：贪心最长匹配

- 从当前位置尝试长度 6→5→4→3→2→1
- 命中 `VALID_SYLLABLES` 就切一段

### 第 3 层：AI 兜底（缩写/无结果）

输入 `"wm"`（声母缩写）：

- 前两层无结果
- 进入 AI：
  - 返回 `["我们","完美","无名", ...]`
  - 延迟大约 ~500ms（取决于模型与网络）

## 质量与速度优化（对话里的进一步结论）

### 1) 不要“每次按键都调 AI”

慢的本质原因通常不是代码，而是：

- 走 AI API（互联网往返）天然 500ms–1s 级别

建议：

- **单个字母/短拼音优先只走本地字典/规则**
- **只有“完全无结果”时才走 AI**（兜底，而不是主路径）

### 2) 让 AI 更像“输入法工具”而不是聊天

对话里提到：如果 prompt 不够像输入法，缩写/网语会翻车（例如 `xdm`）。

做法：

- system prompt 明确角色：输入法候选生成工具
- 在 prompt 里加入少量示例（网语/缩写）能显著提升：
  - `xdm` → `兄弟们`
  - `yyds` → `永远的神`
  - `zqsg` → `真情实感`

### 3) 设备侧做防抖，减少请求频率

小屏输入会“每按一个键就请求一次”，即使本地秒回也会让 UI 很忙。

建议设备侧加防抖：

- 停止输入 300ms 后才发 `/pinyin`

### 4) 如果追求“像手机一样快”：主机侧上离线引擎

对话里给的方向：

- 在主机侧安装离线拼音库/引擎（毫秒级）：
  - Python 侧：`pypinyin`（拼音处理）、`pinyin2hanzi`（拼音→汉字 + 词频排序）
  - 更极致：Google `libpinyin`（C 引擎，速度更强）
- 把声母缩写、长拼音也尽量离线处理
- AI 只处理极少数极端输入

补充：桌面输入法（搜狗/微软）通常是黑盒桌面应用，**不开放“拼音→候选”的 API**，所以要靠开源库/自建服务。

### 5) 联想（/suggest）不一定值得：必要时可以关掉

对话里有一个很现实的取舍：

- 联想如果走 AI，延迟高、准确率一般，会让体验“更慢更烦”
- 如果你的目标是“像电脑输入法一样快”，可以：
  - 只做拼音候选（离线引擎）
  - **关闭联想**，或把联想变成“可选开关/异步不阻塞输入”

## 验收标准

- 常见输入（短拼音/常用词）：
  - 绝大多数请求 < 50ms（局域网 + 本地 Python）
- 兜底输入（缩写/无结果）：
  - 允许 300–800ms，但必须返回可用候选
- 整体：
  - 不因为 AI 超时导致设备端卡死（超时要能返回空/默认候选并提示）
