## 目标

把“云端语音识别”做成可替换部件，避免：

- 换服务商就改一堆业务代码
- 音频格式不符合要求导致识别失败却难排查
- API Key 直接写进脚本/仓库，后续泄露与换 key 成本高

---

## 前置条件

- 输入音频来源：文件（WAV）或从串口采集后落盘
- PC 侧具备基础音频处理能力（读采样率/声道/时长，必要时重采样到 16kHz）

---

## 设计边界

- 适配器只负责“识别”，不负责 VAD/降噪/对话逻辑
- 适配器输出统一文本；上层决定是否保存到文件、是否带时间戳、是否做多轮对话

---

## 建议接口（稳定最重要）

定义一个最小接口即可（伪代码）：

```python
class ASRProvider:
    def recognize_wav(self, wav_path: str, *, language: str = \"zh\") -> str:
        ...
```

关键点：

- **统一输入**：优先 WAV 文件路径（方便 debug 与复现）
- **统一输出**：纯文本（必要时额外返回置信度/分段，但要明确 schema）
- **统一错误**：抛异常或返回错误码，但必须可观测（不要静默返回空字符串）

---

## 音频格式检查（强烈建议）

在调用云端前做一次本地检查并打印关键信息：

- 采样率（多数服务对 16kHz 支持最好）
- 声道（mono/stereo）
- 位深（16bit 常见）
- 时长（避免超长导致服务端超时/计费异常）

如果服务商要求固定格式，建议在适配器内做“自动重采样到 16kHz mono 16bit”，并把“是否发生了转换”写入日志。

---

## 密钥管理（必须）

对话中出现过“需要用户自行填写 API 密钥”的模式，建议工程化落地为：

- 本地 `.env` / `config.local.json`（不入库）
- 仓库只提供 `config.example.*` 模板与字段说明
- 运行时从环境变量加载（如 `ASR_TENCENT_SECRET_ID` 等）

并且：

- 永远不要把 Secret 直接写进脚本
- 把“缺 key 的报错”做得足够清晰（明确缺哪一个字段、去哪里申请）

---

## 验证顺序

1. 用一段短 WAV（1～3 秒）跑通单一服务商
2. 加入自动格式转换（如果需要）
3. 再接入第二家服务商，验证切换不影响上层逻辑
