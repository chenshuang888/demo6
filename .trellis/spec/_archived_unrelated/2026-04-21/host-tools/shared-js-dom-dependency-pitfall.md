# Pitfall：共享 JS 文件的 DOM 依赖导致页面加载即崩溃

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-OLED-----.md`）“共享 JS 文件中的 DOM 依赖（关键风险点）”。
>
> 典型背景：一个仓库里有 uploader 页面、store 页面等多个 HTML；为了复用，把 `serial.js/protocol.js/ui.js` 做成共享脚本。

## 一句话结论

**共享脚本必须“可在任意页面被加载而不崩”：任何 DOM 引用都要么延迟到函数调用，要么做空值容错。**

---

## 触发条件

- `serial.js` / `ui.js` 在脚本加载阶段就执行 `document.getElementById(...).addEventListener(...)`
- 某些页面（例如 store.html）不包含这些元素
- 结果：页面一加载就抛异常，后续逻辑全部中断

---

## 现象（可观测信号）

- 页面白屏或功能不可用
- 控制台报错：`Cannot read properties of null (reading 'addEventListener')`（或类似）
- 看起来像“串口不工作”，但其实脚本初始化就挂了

---

## 根因

对话中点名的高风险点：

- `serial.js` 硬编码依赖：
  - `#portSelect`（串口选择）
  - `#connectBtn`（连接/断开）
- 且存在 **立即执行**的事件绑定（脚本加载时就绑定）
- `log()` 函数引用 `#logContainer`，而商店页可能没有该元素；但 `log()` 又被多个脚本广泛调用 → 形成“全局炸点”

---

## 修复（最小改动优先）

1. **对 `log()` 做容错**：
   - 如果 `#logContainer` 不存在，fallback 到 `console.log`（保证不抛异常）
2. **把脚本初始化阶段的 DOM 绑定改为可选**：
   - 找不到元素就跳过绑定（或把绑定延迟到页面 ready 后再执行）
3. **页面侧保证必需元素存在**（若某页面确实需要串口功能）
   - uploader/store 按各自需求提供对应 DOM

---

## 预防（让它不再发生）

- 规则：共享脚本里禁止“无判空的顶层 DOM 访问”
- 结构：把 UI 相关绑定集中到页面入口文件（每个页面一个 bootstrap），共享脚本只提供纯函数

---

## 验收标准

- 任意页面加载共享脚本都不报错（即使页面不提供串口相关 DOM）
- store 页面不具备串口 UI 时，仍能正常工作（不会被 log() 或 serial.js 初始化拖死）
