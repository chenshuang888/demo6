## 目标

让“PC 端实时串口下发 PCM → 模块端稳定流式播放”具备工程级稳定性：

- 不丢数据（或丢数据可观测且可控）
- 不卡顿（或卡顿可归因：发送端节奏 / 接收端缓冲 / 播放端缺粮）
- 尽可能减少点击声/沙哑（DMA/边界不连续导致）

- 固件侧启用 OUTSIDE（外部数据源）播放路径（API 名称以 SDK 为准）
- 串口下行协议：`spec/iot/protocol/pcm-serial-stream-contract.md`

## 设计边界（关键）

- OUTSIDE 播放属于**生产者-消费者**模型：串口接收线程生产、SDK 播放线程消费
- 不要把“串口读 + SDK 喂数 + 播放控制”全塞进一个任务里，否则必然在边界条件下抖动/丢数/卡顿

---

## 推荐结构（稳定架构）

### 1）双任务 + 环形缓冲

- `serial_recv_task`（高优先级）
  - 读取串口数据
  - 写入 FreeRTOS `StreamBuffer`（环形缓冲）
  - 目标：**优先保证串口不丢**
- `stream_feed_task`（中优先级）
  - 从环形缓冲读取数据
  - 按固定 chunk 喂给 OUTSIDE/SDK
  - 目标：保证播放侧“持续供粮”

对话中的参考参数（可作为起步值）：

- 环形缓冲：`100KB`（覆盖几秒抖动）
- 启播预缓冲：`32KB`（约 1 秒，覆盖起播阶段抖动）
- 喂数 chunk：`512B`

### 2）握手语义（必须对齐协议）

模块回 `0xAA` 前必须确保：

- 环形缓冲已创建
- 串口接收任务已启动（至少已具备立即搬运数据的能力）

否则 PC 收到 `0xAA` 立刻爆发式发送时，会在“接收任务尚未启动/尚未进入调度”窗口里把串口缓冲打爆。

## 点击声/沙哑的典型根因与修复

对话中可复现的根因链：

- DMA/双缓冲边界处波形被硬切 → 相位突变 → pop/click
- 某些频率更容易暴露（与块边界/周期关系有关）

修复建议（尽量不破坏协议）：

1. **发送端对齐**：PC 端把 PCM 总长度对齐到固定粒度（对话中出现 `256 samples`）
2. **发送端淡出**：对补齐的尾部做线性淡出到 0
3. **接收端不做二次切片**：尽量保持 chunk 边界稳定，避免出现“短块+长块”的抖动模式

## OUTSIDE 播放的“中断当前播放并切到 OUTSIDE”的实现建议

对话中给出的结论：

- SDK 不一定提供“单 API 一步中断并切换”的现成接口
- 但可以按标准流程组合实现：
  1. `stop_play(...)`
  2. 等待播放器进入 IDLE（要有超时保护）
  3. `set_curr_outside_handle(stream, end_sem)`
  4. `play_with_outside(...)`

> 重点不是函数名，而是“停→等状态稳定→切句柄→启播”的时序。

## 验证顺序

1. 先验证“纯播放”最小闭环（PC 端发短音频，模块回 `0xBB`）
2. 再验证长音频（>30 秒）不卡顿
3. 最后引入“中断播放/切换播放源”的场景，观察是否出现死锁/状态机卡住

---

## 常见问题（快速排障）

- 卡顿但不丢串口：优先看 feed_task 是否被抢占（优先级/栈/临界区）
- 丢串口数据：优先看 recv_task 优先级与串口环形缓冲容量
- 播放有点击声：优先看对齐/淡出/块边界一致性
