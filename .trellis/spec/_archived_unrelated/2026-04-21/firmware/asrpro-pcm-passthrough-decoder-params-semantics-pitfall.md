## 一句话结论

**很多“PCM 直通/解码”API 的 `remain_size/out_size` 是“输入可用量 + 输出消耗结果”的复用参数：把它当成单向只读会直接导致无声或数据错乱。**

---

## 触发条件

- 使用 SDK 提供的 PCM passthrough / decode API
- API 形参存在类似：
  - `*p_remain_size`（输入/输出复用）
  - `*p_out_size`（输出）
- 上层错误理解这些参数的读写方向与语义

---

## 现象（可观测信号）

- 完全无声（上层以为“写进去了”，但解码器认为可用数据为 0）
- 播放杂音/错乱（remain/out 的边界处理出错，导致读写指针乱跳）

---

## 根因（为什么会这样）

对话中的语义澄清（必须按此理解）：

- `*p_remain_size`（输入）：本次解码可用的实际数据量（例：1152 bytes）
- `*p_out_size`（输出）：本次实际输出的字节数
- `*p_remain_size`（输出）：未消费的剩余数据量（通常为 0，也可能 >0）

一旦把 `*p_remain_size` 当成只读，或忽略它的“输出含义”，上层就无法正确推进数据窗口。

---

## 修复（最小改动优先）

1) 严格按“输入/输出复用”语义更新 `*p_remain_size`
2) 上层循环以“消耗了多少”驱动，而不是以“我写了多少”驱动
3) 每次调用后记录：
   - 入参 remain
   - 出参 out_size
   - 出参 remain
   用于快速定位是否存在卡住或负进展

---

## 预防（让它不再发生）

- 把该 API 的参数语义写进工程内的“音频播放契约”或 wrapper 注释里
- 对 wrapper 增加断言：
  - `out_size <= in_remain_size`
  - `out_remain_size <= in_remain_size`

---

## 验收标准

- 同一份 PCM 数据输入：
  - 修复前：出现无声/错乱
  - 修复后：能稳定输出，且 remain/out_size 随时间单调推进，不出现长期卡住
