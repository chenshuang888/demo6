## 一句话结论

我认可 manifest 思路，但 **不建议第一步就做“运行时把 JSON 交给 `resource_manager` 解析”**。先把“资源消费解耦”做好，再逐步把 locator/定位收回框架层。

---

## 你现在不应该做的事（常见踩坑）

如果现在立刻要求“框架自动决定所有资源落盘位置”，你必须提前定很多东西：

- key → path 的映射规则
- 各资源类型的目录策略
- app 私有目录结构
- 公共资源命名空间结构

这会把问题一下子做复杂、并且容易定错（后续改动成本很高）。

---

## 三选一方案（按推荐顺序）

### 方案 A（最推荐）：静态 C 资源注册表

每个 app 提供一份静态清单：

```c
static const resource_decl_t s_demo_resources[] = {
    ...
};
```

并在 app 注册时提交给 `resource_manager`。

优点：

- 最简单、最稳定
- 编译期检查强
- 运行时无 JSON 解析负担

### 方案 B：构建期 JSON → C（生成 `.c/.h`）

开发时写 JSON，构建脚本生成 C/H，再编译进固件。

优点：

- 保留“清单书写体验”
- 运行时依然轻量

缺点：

- 需要补工具链/脚本

### 方案 C：运行时直接解析 JSON（最不推荐，放最后）

建议最后再考虑：复杂度高（解析、容错、性能、版本兼容、回滚策略），并且容易把架构带偏到“重平台”。

---

## 推荐推进顺序（最小风险）

1. 先把消费解耦做好：对外用资源 key（语义名）访问，不暴露路径/目录/格式。
2. 第一阶段允许注册时携带 locator（定位信息）：
   - 目的是让系统先跑起来，不逼迫一次性定死所有映射规则。
3. 第二阶段再逐步把 locator 抽象掉：
   - 把定位规则沉到 `resource_manager` 的内部实现中。

---

## 验收标准

- 第一版不引入运行时 JSON 解析。
- app 层对资源的依赖是“key/语义”，不是“路径/目录结构”。
- 后续即使资源目录结构变化，app 代码不需要跟着改（只改 `resource_manager`/loader 层）。
