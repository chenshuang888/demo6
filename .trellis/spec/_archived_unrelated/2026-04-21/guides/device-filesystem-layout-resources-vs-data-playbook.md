# Playbook：设备端文件系统布局——静态资源 vs 运行时数据（`/res` 与数据分区分离）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo5.md`）中关于资源系统与目录规划的关键结论：先分清“静态资源”和“运行时数据”，不要把两类东西混进同一套目录/同一套 API；第一版不要急着上 manifest。
>
> 目标：给 IoT 项目一个清晰的落地布局，让资源（字体/图片/UI 资产）与运行时数据（配置/缓存/下载/日志）分离，后续 OTA/扩展更稳。

## 一句话结论

- **静态资源**：随固件版本交付（构建期打包进分区），优先挂载到 `/res`
- **运行时数据**：设备运行中产生/变更（缓存、下载、配置、日志），应放到独立的数据分区（例如 `/data`），不要混进 `/res`

---

## 一、先分清两种东西（否则后面一定乱）

### 1) 静态资源（Static Assets）

典型特征：

- 构建期产出（随镜像一起烧录）
- 版本化、可复现
- 适合放：字体、图片、icon、固定 UI 资源、内置配置模板等

推荐挂载点：`/res`

### 2) 运行时数据（Runtime Data）

典型特征：

- 运行中产生/更新
- 可能需要清理、回收、迁移
- 适合放：缓存、下载包、用户配置、日志、数据库等

推荐挂载点：`/data`（或按你项目习惯命名）

---

## 二、我建议的目录结构（示意）

### 资源目录（静态）

```text
/res/
  public/
  apps/
    <app_id>/
```

> `public` 与 `apps/<app_id>` 的分法对“多 app 资源归属”更稳定。

### 数据目录（运行时）

```text
/data/
  cache/
  logs/
  downloads/
  config/
```

---

## 三、第一版不要做的事（对话里的强建议）

- 不要让 launcher/app 直接“拼路径”访问资源
- 第一版不要上来就搞 manifest 系统（先把 key/边界跑通）

第一版最简单可落地方案是：

- UI 只用 role/key
- manager 负责把 role/key 映射到路径（或其他后端）
- 物理目录结构可以先朴素，后续再演进

如果以后要升级，再加 manifest（按 app 声明资源集合），让 `app_manager` 在切换时触发 preload/release。

---

## 验收标准

- `/res` 与 `/data` 语义清晰，不混用
- app 层不直接访问路径（无 `/res/...` 硬编码）
- 资源目录结构未来可调整，但 app 代码不需要跟着改
