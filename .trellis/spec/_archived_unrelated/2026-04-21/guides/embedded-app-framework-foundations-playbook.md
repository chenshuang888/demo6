## 一句话结论

你缺的不是更多 `manager.c`，而是缺 **少量一阶框架（foundation）** 来约束 app 的写法；优先把这些 foundation 抄到位，app 才不会越写越乱。

---

## ESP-IDF 上最划算的三件“一阶框架”

1. **事件总线（Event Bus）**
   - 目标：所有 app 动作都变成 `APP_ACTION / SYS_EVENT`，禁止 app 之间直接互相调用/乱发消息。
2. **页面栈 / 路由（Navigation / Scene Stack）**
   - 目标：app 只写“页面”，modal/tab/返回等导航由统一栈管理，避免每个 app 自己发明一套。
3. **设置与持久化（KV Store + 少量文件）**
   - 目标：app 不再自定义一堆路径/版本迁移；统一为“按 `app_id` 的 KV + 少量文件”。

---

## 关键原则：抄“接口与分层”，不要直接抄“实现代码”

- 抄代码很容易把别人的假设/硬件/历史包袱一起抄进来
- 抄接口与分层，才能让你的项目保留自己的节奏与约束

---

## “像 LVGL 一样好移植”的正确理解

好的框架确实应尽量“不看硬件”，但前提是它把“看硬件”的部分压缩到一个非常薄、非常清晰的移植层。  
也就是说：不是完全不需要硬件信息，而是 **硬件依赖被隔离并最小化**。

## 最小“底座模块”清单（你说的“把 LVGL 留下”至少要包含这些）

对话里给出的一个可执行切分是：把工程分解为 8 块固定能力（名字可调整，职责别混）：

1. 启动主链路：`main` → UI 线程循环
2. 驱动与 LVGL 对接层：`drivers/`（lcd/touch/backlight… + lvgl glue）
3. VFS 挂载与原始文件能力：例如 `vfs_service.*`
4. 资源系统：`resource_manager.*`（对外按 key，屏蔽 `/res` 细节）
5. 存储系统：例如 `storage_manager.*`（运行时数据 `/data` 等）
6. App 生命周期与切屏：`app_manager.*`
7. UI/Logic 分离与 IPC：例如 `app_runtime_engine.*`
8. 统一应用入口与 Launcher：`apps_entry.*` + `launcher_app.*`

> 目的：你可以删掉 demo/临时代码，但这些“底座能力”不能删，否则“留下 LVGL”只是留下 UI 库而不是系统能力。

---

## 推荐落地顺序（不难，也不把框架搞重）

1. 先把 IPC 载荷变成通用协议（清掉 manager 里的 app 专属字段污染）
2. 做一个 `app_template`（或 `sample_app`），跑通：按钮 → logic（耗时）→ UI 更新
3. 把现有 `launcher_app` 迁成同样结构（哪怕 logic 很薄，也要“每个 app 都带 logic”的一致性）

---

## 验收标准

- 新增一个 app 时，你能明确选择：它触发的是 event bus、navigation 还是 settings，而不是直接互调。
- UI 线程没有被网络/存储等耗时 IO 阻塞（耗时走 logic/后台，再回 UI 更新）。
- 删除 demo 后，工程仍保留“可跑的底座能力清单”（上面 8 块）。
