## 一句话结论

**文件系统用“资源类型优先”的目录保持整洁；UI 层只使用 `RESOURCE_ID`（例如 `FONT_ID_* / IMAGE_ID_*`），由中间层做映射与加载。**

---

## 第一版范围怎么收敛（避免过度设计）

对话里强调：第一版可以非常克制——先只服务 LVGL，别为了“未来可能有音频/视频”把复杂度提前引入。

- 当前阶段 `resources/`（或 `assets/`）可以直接理解为：**LVGL 资源目录**
- 现阶段只需要三类：字体、图片、图标
- UI 层坚持：**不直接碰文件路径**（资源落盘/分区/挂载是实现细节）

目录形态上有两种都合理（按你后续是否要扩其它资源选择）：

```text
resources/
  fonts/
  images/
  icons/
```

或：

```text
resources/
  lvgl/
    fonts/
    images/
    icons/
```

---

## 快速判断：某个资源该放哪

如果你不知道某个资源应该归到字体还是图片，先问一句：

**它在 UI 里扮演的是“文字”还是“图形”？**

- 更像“文字”（可按字号渲染、可换颜色、跟文本排版走）：优先进 `fonts/text`
- 图标字体（本质仍是字体子集）：进 `fonts/icon`
- 位图图片：进 `images`

> 这条规则的目的不是“强行统一”，而是让目录永远按“资源类型”分层，不要中文/英文/图标/图片平铺混在一起。

## 先做对两件事（别一上来做大框架）

对话里反复强调的平衡点是：

- 你确实需要“资源系统”，否则后面会乱
- 但第一版不要追求“框架一次到位”，先把 **分类** 与 **解耦** 做对

## 目录分层建议（先按资源大类，再按家族/子类/尺寸）

推荐形态（示例）：

```text
assets/
  fonts/
    text/
      zh/
        noto_sans_sc/
          12_regular.bin
      en/
        montserrat/
          12_regular.bin
    symbol/
      ui/
        12_regular.bin
  images/
    ui/
      wifi_16.bin
    brand/
      logo.bin
```

关键规则：

1. 第一层必须按 **资源类型**
   - `fonts/` 与 `images/` 不要混
2. 同类型资源再按“家族/子类/尺寸”细分
   - 字体按语言/字体家族/字号
   - 图片按用途/模块/尺寸

## “特殊符号”怎么处理（两类）

对话给的判断方式：

- 如果属于文字的一部分（度数符号、箭头、数学符号、常用 UI 字符）
  - 放进 `fonts/text/symbol/`（本质仍是字体子集）
- 如果是图形图标（更像图片）
  - 放进 `images/`（不要硬塞进字体系统）

---

## LVGL 里“文字/图标/图片”通常怎么显示（避免概念混乱）

对话里强调：不要把“图标”当成单一实现，它至少有两种常见形态：

### 1) 图标字体（Icon Font）

本质上还是“字形 glyph”，因此在 LVGL 里通常走：

- `lv_label_create()`
- `lv_label_set_text()`
- 再把字体切到 icon role（例如 `FONT_ROLE_ICON`）

适合：

- 箭头、勾叉、状态符号等小图标
- 和文本一起排版（对齐/换行/样式）更自然

### 2) 图片图标（Image Icon）

本质是图片，通常走：

- `lv_image_create()`
- `lv_image_set_src()`

适合：

- logo、插图、头像、装饰图
- 不适合用字体“硬塞”

### 最小可见示例建议（不引入解码器）

对话里给的 bring-up 方式是：先用一个内置 `lv_image_dsc_t` 的小 RGB565 图片块验证 image widget 链路（不依赖 PNG/JPG 解码），等 UI 结构稳定后再接文件系统图片资源。

### 补充：底层并非“毫无关系”

对话里还提醒：虽然上层 API 不同（label vs image），但最终都会进入 LVGL 的绘制流程，画到屏幕缓冲里；差异主要在于“数据来源”与“控件语义”不同。

## 关键契约：UI 层不要关心目录/路径/格式

UI 层应该拿到的是抽象 ID，例如：

- `FONT_ID_ZH_BODY_16`
- `FONT_ID_EN_TITLE_20`
- `FONT_ID_ICON_20`
- `IMAGE_ID_LOGO`
- `IMAGE_ID_WIFI_16`

然后由中间层把这些 ID 映射到真实资源（文件路径、embedded blob、flash 分区、cache 等）。

这样做的直接收益：

- UI 代码不受“资源存放位置/格式变化”影响
- 资源系统可以逐步演进（从最简单的嵌入式数组 → 文件系统 → 分区 → OTA 更新），上层不改

---

## 类比：为什么“UI 不碰路径”是更通用的做法（Android 视角）

对话里给了一个非常直观的类比：

- Android UI 通常通过 `R.drawable.xxx / R.string.xxx / R.font.xxx` 拿资源（本质是 Resource ID）
- 框架/资源系统负责：类型区分、缓存、不同版本适配、主题化
- UI 直接按路径加载（类似 `Typeface.createFromAsset(...)`）不是不能用，但更容易把“资源实现细节”散到 UI 各处

把这个类比映射回 IoT 设备端 UI：**`RESOURCE_ID` 就是你的 `R.xxx`**；你不需要一上来做很重的资源系统，但要尽早把“路径隔离在一处”这条边界立住。

---

## 推荐的资源分层（能力层 / 适配层）

对话里把“更顺的分层”讲得很明确：

1. **能力层（Capability Layer）**
   - 负责“资源能被加载/解析/缓存”的能力
   - 例如：读取字体 blob、创建 `lv_font_t*`、图片解码、缓存策略
2. **适配层（Adapter Layer）**
   - 负责“把能力层包装成 UI 友好的 API/ID”
   - 例如：`asset_get_font(FONT_ID_...)`、`asset_get_image(IMAGE_ID_...)`

> 重要：第一版不要层数太多，保持朴素可用。

---

## 第一阶段建议（只支持最少能力）

对话中强调第一阶段不要贪多，可以先只支持：

- 少量字体（UI 默认小字体 + 标题字体）
- 少量图标/图片（必要的 UI 图标）
- ID 映射表（哪怕是一个 `switch`/数组表）

其余能力（动态加载、复杂 cache、按页面子集自动生成）留到后面迭代。

---

## 验收标准

- 目录不混乱：字体与图片分开，新增资源有固定位置
- UI 代码只出现 `RESOURCE_ID`，不出现路径/文件名/格式细节
- 第一版资源系统可迭代：你可以“先能用”，再逐步加能力，而不会推翻 UI 层
