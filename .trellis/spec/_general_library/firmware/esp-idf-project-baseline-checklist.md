# Checklist：ESP-IDF 新项目基线盘点（拿到空工程时）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo5.md`）开头对“新建 ESP-IDF 项目”进行快速盘点的流程。
>
> 目标：当你刚拿到一个几乎空的 ESP-IDF 工程时，先把“项目真实状态”摸清，避免后续移植/接入（如 LVGL）时方向跑偏。

---

## 一句话结论

**先确认“工程是否仍是模板态 + 入口/组件/环境配置是否就绪”，再开始移植驱动或接 LVGL。**

---

## 1) 工程结构（代码层）

- [ ] 工程名与 `project()` 是否正确（通常在根 `CMakeLists.txt`）
- [ ] 主组件是否存在且注册正常（通常在 `main/CMakeLists.txt`）
- [ ] `app_main()` 是否存在、当前做了什么（通常在 `main/main.c`）
- [ ] 当前是否“模板态”（几乎没有业务代码/外设初始化）
- [ ] 是否已经存在自定义组件目录（如 `components/`、`drivers/`）

---

## 2) 环境与工程化（开发体验层）

> 这一段尤其在 Windows/PowerShell 下容易踩坑：不是代码挂，而是命令链/路径/环境冲突导致“看起来编译不了”。

- [ ] `.vscode/settings.json` 是否已配置 ESP-IDF 路径（如本机 `c:\Users\...\esp-idf`）
- [ ] 是否存在 devcontainer（`.devcontainer/devcontainer.json`）并且与你的预期一致
- [ ] 项目使用命令约定是否明确（例如在 Windows 上统一使用 `python` 而不是 `python3`）

---

## 3) 最小冒烟建议（可选）

> 如果你希望快速确认工程链路可用：做一个“最小打印”即可，不要一开始就接全量 LVGL。
>
> 注意：如果团队协作约定为“本地编译/烧录由用户自己点按钮完成”，那这里仅写建议，不强制执行。

- [ ] `app_main()` 输出一条日志（例如 `ESP_LOGI`）用于验证工程能跑起来
- [ ] 确认 target/board 选择正确（ESP32-S3 等）

---

## 验收标准

- 你能用 1 分钟回答：工程是否模板态、入口在哪、主组件在哪、环境路径是否合理
- 接入 LVGL/移植旧 `drivers/` 前，已经明确：哪些是“板级参数”、哪些是“LVGL glue”、哪些是“纯硬件驱动”
