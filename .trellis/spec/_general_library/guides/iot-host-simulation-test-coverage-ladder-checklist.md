# Checklist：IoT 主机侧模拟测试覆盖阶梯（从“够用”到“极致”）

> 来源：对话蒸馏（`C--Users-ChenShuang-Desktop-esp32-demo3.md`）里对一套主机侧测试（API 契约 + C 行为模拟 + HTTP 分块模拟）的覆盖盘点与差距分析。
>
> 目标：明确“哪些能在电脑上高效覆盖、哪些必须上真机”，避免在错误的地方堆测试。

---

## 已覆盖（电脑上可高性价比模拟的部分）

- [ ] **API 契约**：HTTP 状态码、JSON 字段名/类型/结构
- [ ] **数据边界**：固定缓冲区溢出、字符串截断（含 UTF-8 边界）
- [ ] **网络异常**：超时/500/乱码 JSON/空响应
- [ ] **请求构造**：设备端发出去的 JSON 是否符合预期
- [ ] **状态机/流程**：页面切换、返回路径、完整用户流程（例如搜索→进入→翻页）
- [ ] **并发竞态（基础）**：用 `seq` 防旧响应覆盖
- [ ] **HTTP 分块接收**：模拟 `http_collect_cb` 这类“按 chunk 追加”的逻辑
- [ ] **数据质量**：去广告、排序正确等业务质量断言
- [ ] **性能**：响应时间阈值（例如 < 8s）

---

## 未覆盖（但理论上可以做的增强项）

### 1) 真 C 编译测试（★★★★★）

- [ ] 把关键解析逻辑抽成独立 C 文件，用 `gcc`/MSVC 编译成可执行文件
- [ ] 喂真实响应数据做回归
- 价值：发现 Python 模拟覆盖不到的问题（指针越界、整数溢出、未定义行为）

### 2) 内存泄漏检测（★★★★）

- [ ] 检查 `xTaskCreate` 是否都有对应 `vTaskDelete`
- [ ] 检查 `malloc/heap_caps_malloc` 是否都有对应 `free`
- [ ] 压测循环 100 次，看内存是否持续增长

### 3) 多任务并发安全（★★★★）

- [ ] 模拟快速连续点击导致“任务堆积”
- [ ] 检查是否有 `s_loading` 之类的互斥标记
- [ ] 检查是否存在任务泄漏（创建了但没退出）

### 4) Wi-Fi 断线重连（★★★）

- [ ] 模拟服务器突然关闭/网络中断
- [ ] 检查超时处理、回退 UI、重试策略

### 5) NVS 存储边界（★★★）

- [ ] 模拟 NVS API 的边界/失败分支
- [ ] 检查是否会导致设置损坏或崩溃

### 6) 长时间运行稳定性（★★★）

- [ ] 模拟 1000 次操作循环
- [ ] 检查是否出现内存增长、响应变慢、任务堆积

### 7) UI 层测试（★★）

- [ ] LVGL 对象创建/销毁是否配对
- [ ] 是否有野指针（对象删除后仍使用）
- 说明：通常需要 LVGL 模拟器或更强的静态分析工具

---

## 选择建议（别为了覆盖率而覆盖）

- 想追求 95%+：
  1. 真 C 编译测试
  2. 内存泄漏检测
  3. 并发/任务堆积压测
- 当前够用（约 85%）：
  - 通信/解析/边界已覆盖到位
  - 剩下的多是“真机才暴露”的问题（Wi-Fi/UI/长时间运行）
  - 建议：先上真机跑，再针对暴露的问题补测试
