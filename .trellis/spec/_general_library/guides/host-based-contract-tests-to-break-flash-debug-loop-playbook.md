## 一句话结论

IoT 的很多 bug 不在“硬件”，而在 **序列化/传输/反序列化**：把它们搬到主机侧做自动化测试，真机只保留 UI/触摸/时序等不可模拟的验证。

---

## 适用范围

- ESP32 ↔ 主机之间用 HTTP/JSON 通信
- 设备端用 cJSON 解析
- 设备端存在固定长度缓冲/`strncpy` 截断等“边界条件”

---

## 推荐三层测试（对话里的覆盖矩阵）

### A) API 层测试（`test_novel.py` 这类）

目标：验证服务端“契约”本身正确。

- HTTP 状态码
- JSON 字段存在性
- JSON 字段类型
- 边界输入（空/无效/越界页码）
- 数据质量（例如去广告、排序正确）
- 用户流程完整性
- 响应时间

### B) C 行为模拟测试（`test_esp32_sim.py` 这类）

目标：即使服务端契约正确，也要验证“设备端写法”是否安全。

典型模拟点：

- `strncpy` 的截断行为（并确保 UTF-8 边界完整）
- 固定缓冲区大小 vs 真实响应字节数（提前发现溢出）
- HTTP 分块接收回调（模拟 `http_collect_cb` 这类“能装多少装多少”的逻辑）
- 设备端解析路径（字段取值、数组遍历、错误处理分支）

### C) 更深覆盖（可选）

当你发现某个模块特别容易出坑（例如小说正文页/大 JSON），再加一层更深的模拟测试（对话里提到 `test_esp32_deep.py` 这类）。

---

## 为什么这对 IoT 特别有效

对话中的关键总结是：

IoT 通信链路本质就是：

- **序列化（JSON）**
- **传输（HTTP）**
- **反序列化（cJSON）**

这三段都能在电脑上复现与断言；真机只需要验证：

- UI 渲染与触摸交互
- 真实网络波动
- 真实内存碎片与长时间稳定性

---

## 验收标准

- 新增/修改服务端字段时：
  - API 层测试能立刻失败并提示“契约变了”
  - C 行为模拟能捕获“缓冲区不足/UTF-8 截断/解析路径崩溃”
- 典型修复节奏变成：
  - `python test`（10 秒）→ 修 → 再 test → 全绿 → 再烧录真机验证
